10:T4526,<h2>서론: 왜 지금 AI 가속기 아키텍처인가</h2>
<p>2026년 5월, 반도체 시장에서 가장 뜨거운 키워드는 단연 <strong>AI 칩</strong>입니다. 삼성전자와 SK하이닉스가 급등하고, 글로벌 AI 반도체 투자가 확장되면서 '모든 것이 NVIDIA GPU로 해결된다'고 여기는 시대를 끝내고 있습니다. 그러나 대규모 AI 모델 학습과 추론의 요구사항은 계속 달라지고 있으며, 그에 따라 <strong>결합된 GPU 아키텍처를 넘어서는 새로운 설계 패러다임</strong>이 필요합니다.</p>
<p>본 글에서는 AI Accelerators 시대의 반도체 설계 핵심 개념과 실제 시스템 설계에 어떻게 적용하는지를 다루겠습니다.</p>

<h2>1.异형 컴퓨팅(Heterogeneous Computing)의 부상</h2>
<h3>전통적 접근의 한계</h3>
<p>단일 نوع의 프로세서(예: CPU only 또는 GPU only)로 모든 워크로드를 처리하는 것은 더 이상 효율적이지 않습니다. 특히 AI 추론에서는 아래와 같은 요구사항이 공존합니다:</p>
<ul>
<li><strong>신경망 추론</strong>: 행렬 곱셈 중심, 대량 병렬 연산</li>
<li><strong>데이터 전처리/후처리</strong>: 제로-copy, 캐시 친화적 접근</li>
<li><strong>제어 플로우</strong>: 분기 예측, 예외 처리</li>
</ul>
<h3>핵심 개념: Domain-Specific Architectures (DSA)</h3>
<p>DSA는 특정 도메인에 최적화된 명령어 집합과 메모리 구조를 가진 프로세서를 의미합니다. Google의 <strong>TPU</strong>, Graphcore의 <strong>IPU</strong>, Cerebras의 <strong>Wafe Engine</strong>이 대표적인 사례입니다.</p>
<pre><code class="language-python"># DSA vs General Purpose: Conceptual latency comparison

<h2>2. 메모리 대역폭 병목 해결: HBM에서。CXL까지</h2>
<h3>문제 정의</h3>
<p>AI 가속기의 성능을 결정하는 핵심 요소는 <strong>연산 throughput</strong>이 아니라 <strong>메모리 대역폭</strong>입니다. 많은 AI Accelerators가 연산 유닛보다 메모리 대역폭에 의해 제한됩니다(계산 폭발, compute-bound보다 memory-bound).</p>
<h3>HBM (High Bandwidth Memory)의 한계와 대안</h3>
<p>HBM은 현재 AI 가속기의 표준 메모리 솔루션이지만, 몇 가지 근본적 한계가 있습니다:</p>
<ol>
<li><strong>엄청난 전력 소모</strong>: HBM3은 칩당 20~30W의 에너지를 소비</li>
<li><strong>비용</strong>: 대형 GPU당 HBM만 $2,000~3,000</li>
<li><strong>형평성 문제</strong>: 멀티 칩 시스템에서 메모리 공유가 어려움</li>
</ol>
<h3>CXL (Compute Express Link): 차세대 메모리 인터커넥트</h3>
<p>CXL은 CPU, GPU, 가속기 간的高速 메모리 공유를 가능하게 하는 호환성 레이어입니다. PCIe 기반으로, 코히irent 메모리 액세스를 지원합니다.</p>
<pre><code class="language-c">// CXL Memory Pool 개념: simplified pseudocode
// 실제 구현에서는 OS kernel module과 hardware abstraction이 필요합니다

typedef struct {
    uint64_t base_addr;
    uint64_t size_gb;
    enum { CXL_MEM, DDR, HBM } memory_type;
    bool coherent_with_cpu;
} MemoryRegion;

class CXLMemoryPool {
private:
    std::vector&#x3C;MemoryRegion> regions;
    size_t total_capacity_gb;

public:
    void register_device(uint64_t bar_addr, uint64_t size, 
                         bool supports_coherency) {
        MemoryRegion region = {
            .base_addr = bar_addr,
            .size_gb = size / (1024*1024*1024),
            .memory_type = supports_coherency ? CXL_MEM : DDR,
            .coherent_with_cpu = supports_coherency
        };
        regions.push_back(region);
        total_capacity_gb += region.size_gb;
    }

    void* allocate_for_accelerator(size_t size, 
                                   bool prefer_cxl) {
        // CXL 메모리가 있다면它在 (preferred)
        if (prefer_cxl) {
            for (auto&#x26; r : regions) {
                if (r.memory_type == CXL_MEM &#x26;&#x26; r.size_gb * 1024 >= size)
                    return (void*)r.base_addr; // simplified
            }
        }
        // fallback: any available memory
        return nullptr;
    }
};
</code></pre>
<p><strong>핵심 포인트</strong>: CXL을 활용하면 GPU 메모리 용량 제한을 극복하고, 멀티 가속기 간大型 모델을 효율적으로 공유할 수 있습니다.</p>

<h2>3. Chiplet 설계: 모듈러 아키텍처의實現</h2>
<h3>Monolithic vs Chiplet</h3>
<p>기존 GPU 설계는 Monolithic die였지만, 대형 AI 가속기는 공정 기술의 물리적 한계와 경제성 문제로 Chiplet 방식으로 전환하고 있습니다.</p>
<pre><code>┌─────────────────────────────────────────────────────┐

│  │              Large Silicon Die                 │  │

└─────────────────────────────────────────────────────┘

<h3>설계 원칙: AMD Instinct MI300X 사례研究</h3>
<p>AMD MI300X는 대표적인 Chiplet AI 가속기입니다:</p>
<ol>
<li><strong>4-way Compute Chiplets</strong>: 5nm 공정의 계산 유닛을 4개 통합</li>
<li><strong>3D V-Cache</strong>: Compute Die 위에 DRAM을 3D 스택킹</li>
<li><strong>Infinity Fabric</strong>: Chiplet 간高速 인터커넥트 (900 GB/s)</li>
</ol>
<pre><code class="language-python"># Chiplet 기반 시스템의 설계 고려 사항: 
# 대규모 언어 모델을 여러 가속기에 분할하는 방법

class ModelParallelism:
    """
    Transformer 모델의 레이어를 여러 AI 가속기에 분할하는 전략
    """
    
    def plan_sharding(self, model_total_params_b, 
                      num_accelerators, memory_per_accelerator_gb):
        """
        모델을 accelerator에 분할하는 전략을 계획합니다.
        
        Args:
            model_total_params_b: 총 파라미터 수 (Billions)
            num_accelerators: 가속기 수
            memory_per_accelerator_gb: 가속기당 메모리 (GB)
        
        Returns:
            sharding_strategy: 분할 전략 설명
        """
        params_per_gb = 2 * (10**9)  # FP16 기준 2GB per Billion params
        
        # Check if model fits in single accelerator
        required_gb = model_total_params_b / params_per_gb
        
        if required_gb &#x3C;= memory_per_accelerator_gb:
            return "Single device deployment possible"
        
        # Calculate tensor parallelism degree
        max_tensor_parallel = num_accelerators
        for tp_degree in range(num_accelerators, 0, -1):
            per_device_memory = required_gb / tp_degree
            if per_device_memory &#x3C;= memory_per_accelerator_gb * 0.85:  # 85% safety margin
                return (f"Tensor Parallelism with TP={tp_degree}\n"
                        f"Per device memory: {per_device_memory:.1f} GB")
        
        # Fall back to pipeline parallelism
        return (f"Pipeline Parallelism required\n"
                f"Model too large for tensor parallelism alone")

# Example: 70B parameter model on 8x H100 (80GB each)
planner = ModelParallelism()
strategy = planner.plan_sharding(
    model_total_params_b=70,
    num_accelerators=8,
    memory_per_accelerator_gb=80
)
print(strategy)
</code></pre>

<h2>4. Hardware-Software Co-design: 설계의 新基準</h2>
<p>###传统적 방법의 문제</p>
<p>하드웨어를 먼저 설계하고, 나중에 소프트웨어를 포팅하는 방식은 AI 시대에 다음과 같은 문제를 야기합니다:</p>
<ol>
<li><strong>시장 진입 지연</strong>: 제품 출시 시 소프트웨어 생태계가 미성숙</li>
<li><strong>비효율적 자원 활용</strong>: 하드웨어 기능을 소프트웨어가 활용하지 못함</li>
<li><strong>높은 비용</strong>: 설계 변경이 제조 후에는 불가능</li>
</ol>
<h3>Co-design 접근법</h3>
<p>현대 AI 가속기 설계는 하드웨어와 소프트웨어를 동시에 설계합니다:</p>

<h2>5. 시스템 설계에 대한 시사점</h2>
<p>AI 가속기 아키텍처의 발전은 시스템 설계자에게 다음과 같은 시사점을 줍니다:</p>
<h3>5.1 추상화 레이어의重要性</h3>
<p>AI 가속기의 다양성을 고려할 때, 하드웨어 의존성을 최소화하는 추상화 레이어가 필수적입니다:</p>
<pre><code class="language-python"># AI Accelerator Abstraction Layer 예시

from abc import ABC, abstractmethod
from typing import List, Optional, Dict, Any
import numpy as np

class AIAcceleratorBackend(ABC):
    """AI 가속기 백엔드 추상화"""
    
    @abstractmethod
    def allocate_tensor(self, shape: tuple, dtype: np.dtype) -> 'AcceleratorTensor':
        pass
    
    @abstractmethod
    def matmul(self, a: 'AcceleratorTensor', b: 'AcceleratorTensor') -> 'AcceleratorTensor':
        pass
    
    @abstractmethod
    def synchronize(self) -> None:
        pass

class AcceleratorTensor:
    """추상화된 텐서 표현"""
    def __init__(self, backend: AIAcceleratorBackend, handle: Any):
        self.backend = backend
        self.handle = handle

class CUDABackend(AIAcceleratorBackend):

<h3>5.2 메모리 계층 구조 설계</h3>
<p>AI 시스템에서 메모리 계층 구조를 잘 설계하는 것은 성능에 결정적입니다:</p>
<pre><code>L0: Register File (AI accelerator 내) - &#x3C; 1KB, 1-cycle
L1: On-chip SRAM - ~10MB, 3-5 cycles
L2: Off-chip HBM - ~80GB, 100-300 cycles
L3: CXL Memory Pool - TB scale, 150-400 cycles
L4: NVMe SSD - TB scale, 100,000+ cycles
L5: Object Storage (S3) - PB scale, async
</code></pre>
<h3>5.3 장애 복구 및弹性</h3>
<p>AI 가속기는 높은 전력 소모와 발열로 인해 장애가 발생할 수 있습니다. 시스템 설계 시:</p>
<ol>
<li><strong>Checkpoint frequency</strong>: GPU당 1~5분마다 모델 상태 저장</li>
<li><strong>Pipeline parallelism</strong>: 단일 가속기 장애 시 다른 가속기가 계속 작업 가능</li>
<li><strong>Graceful degradation</strong>: 전체 시스템 기능 유지를 위한 부분적 동작 모드</li>
</ol>
<hr>
<h2>결론: Architect가 알아야 할 5가지 핵심</h2>
<ol>
<li>
<p><strong>GPU만으로는 충분하지 않다</strong>: AI 워크로드의 다양성을 고려하면, TPU, NPU, DSA 등 다양한 가속기를 활용하는 것이 필수</p>
</li>
<li>
<p><strong>메모리 대역폭이 병목이다</strong>: 연산 throughput보다 메모리 접근 패턴을 최적화하는 것이 더 큰 성능 향상</p>
</li>
<li>
<p><strong>Chiplet 설계가 새로운 표준이다</strong>: 단일 대형 칩보다 모듈러 접근이 비용과 수율 측면에서 유리</p>
</li>
<li>
<p><strong>Hardware-Software Co-design이 필수다</strong>: 하드웨어와 소프트웨어를 분리해서 설계하던 시대는 끝났습니다</p>
</li>
<li>
<p><strong>추상화가 경쟁력이다</strong>: 특정 가속기에 의존하지 않는 추상화 레이어를 설계해야 기술적 의사결정의 유연성을 확보합니다</p>
</li>
</ol>
<p>AI 가속기 분야의 빠른 변화 속에서, 이러한 핵심 개념을 이해하고 적용하는 것이 시스템 설계자에게 필요한 새로운 역량입니다.</p>
