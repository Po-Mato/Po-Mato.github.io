<h2>1) React 19: ‘클라이언트에서 다 한다’에서 ‘경계 설계’로</h2>
<p>요즘 팀에서 가장 많이 생기는 문제는 기능 자체보다도 <strong>어디에서 렌더링/데이터를 처리해야 하는지</strong>(Server vs Client)에서 나온다.</p>
<h3>체크 포인트</h3>
<ul>
<li>상태/이벤트가 필요 없는 UI는 <strong>서버 컴포넌트(RSC)로 최대한 밀어 넣기</strong>
<ul>
<li>번들 크기↓, TTI 개선, hydration 부담 감소</li>
</ul>
</li>
<li>클라이언트 컴포넌트는 다음을 기준으로 최소화
<ul>
<li>폼 입력, 드래그/애니메이션, 브라우저 API, 실시간 인터랙션</li>
</ul>
</li>
<li>“렌더링은 되는데 느린” 유형이면
<ul>
<li>(1) 클라이언트 번들 크기</li>
<li>(2) hydration 비용</li>
<li>(3) 비동기 데이터/서스펜스 경계(Suspense boundary)
를 먼저 의심</li>
</ul>
</li>
</ul>
<h3>실무 팁</h3>
<ul>
<li><code>"use client"</code>는 <strong>파일 단위 전염</strong>이라, 작은 인터랙션 때문에 큰 트리 전체가 클라로 넘어가지 않게 컴포넌트 분리.</li>
</ul>
<hr>
<h2>2) Next.js 16 + App Router: 데이터 패칭/캐싱이 ‘설계’가 됨</h2>
<p>App Router에서는 “fetch 한 번”이 아니라 <strong>캐시 정책이 곧 동작</strong>이다.</p>
<h3>체크 포인트</h3>
<ul>
<li>페이지/세그먼트가 “언제 업데이트되어야 하는지”를 먼저 결정
<ul>
<li>완전 정적: <code>export const dynamic = "force-static"</code></li>
<li>특정 주기로 재생성: ISR(<code>revalidate</code>) 전략</li>
<li>사용자별/실시간: <code>force-dynamic</code> 또는 캐시 제어</li>
</ul>
</li>
<li><code>fetch</code> 호출은 기본적으로 캐시가 걸릴 수 있으므로
<ul>
<li>의도에 맞게 <code>cache: "no-store"</code> / <code>next: { revalidate: n }</code> 등을 명시</li>
</ul>
</li>
<li>SEO/공유(OG) 메타는 <code>generateMetadata</code>를 적극 활용</li>
</ul>
<h3>이 블로그 기준으로 바로 적용 가능한 개선 아이디어</h3>
<ul>
<li><code>/portfolio</code>와 <code>/posts</code>에 있는 <code>generateMetadata</code> 패턴을 공통화(유틸)해서 유지보수성↑</li>
<li><code>sitemap.ts</code>에 이미 <code>/portfolio</code> 포함했으니, 다음은 <code>tags</code> 상세도 포함할지 검토</li>
</ul>
<hr>
<h2>3) RSC(React Server Components): “보안/성능”이 같이 좋아지는 구간</h2>
<p>RSC를 쓰면 서버에서만 가능한 작업을 자연스럽게 할 수 있다.</p>
<h3>체크 포인트</h3>
<ul>
<li>민감한 토큰/키는 <strong>서버에서만 사용</strong>하고 클라이언트로 절대 보내지 않기</li>
<li>마크다운/콘텐츠 렌더링은 서버에서 처리하면
<ul>
<li>클라 번들↓</li>
<li>XSS 방어(지금도 <code>rehype-sanitize</code> 적용 중) 논리가 깔끔</li>
</ul>
</li>
</ul>
<hr>
<h2>4) Tailwind v4: 설정/플러그인 호환 이슈를 먼저 점검</h2>
<p>Tailwind v4는 “그냥 업그레이드”라기보다 <strong>빌드 체인/구성 방식</strong>이 바뀐 느낌이라,
레거시 설정을 그대로 들고 가면 예상치 못한 문제가 날 수 있다.</p>
<h3>체크 포인트</h3>
<ul>
<li>PostCSS 설정(<code>postcss.config.mjs</code>)과 Tailwind 패키지 버전 정합성</li>
<li>기존 플러그인/프리셋이 v4를 지원하는지</li>
<li>다크모드, typography(프로즈) 같은 스타일이 기대대로 나오는지</li>
</ul>
<h3>이 블로그 기준</h3>
<ul>
<li>글 본문이 <code>prose prose-invert</code> 기반이라, Tailwind 업데이트 후
<ul>
<li>코드블록/링크/헤딩 색상</li>
<li>리스트/인용문 스타일
이 깨지지 않는지 한 번 확인해두는 게 좋음.</li>
</ul>
</li>
</ul>
<hr>
<h2>5) TypeScript 최신화: “빌드 성공”보다 “DX/안전성”을 챙기는 설정</h2>
<p>Next + TS에서는 타입 안정성뿐 아니라 <strong>빌드 파이프라인/ESM/CJS 경계</strong>가 자주 이슈가 된다.</p>
<h3>체크 포인트</h3>
<ul>
<li><code>tsconfig.json</code>에서
<ul>
<li><code>strict</code></li>
<li><code>noUncheckedIndexedAccess</code></li>
<li><code>exactOptionalPropertyTypes</code>
같은 옵션을 단계적으로 도입하면 장기적으로 사고가 줄어듦</li>
</ul>
</li>
</ul>
