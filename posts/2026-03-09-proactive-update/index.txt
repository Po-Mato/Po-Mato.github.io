10:Tf4f,<h1>Next.js 프로젝트의 기술 부채 청산 전략: Lint부터 Vitest까지</h1>
<p>프로젝트가 성숙해짐에 따라 필연적으로 발생하는 **기술 부채(Technical Debt)**는 적시에 해결하지 않으면 개발 속도를 저하시키고 안정성을 해치는 암초가 됩니다. 최근 본 블로그 프로젝트(<code>Po-Mato/tech-blog</code>)에서 진행한 대규모 부채 상환 과정을 바탕으로, 어떻게 시스템적으로 품질을 강제할 수 있는지 아키텍처 관점에서 분석합니다.</p>
<h2>1. 기술 부채의 식별과 우선순위</h2>
<p>이번 부채 청산 작업에서는 다음과 같은 문제들을 우선순위별로 해결했습니다.</p>
<ol>
<li><strong>품질 게이트의 부재</strong>: Lint 에러가 있음에도 빌드와 배포가 성공하는 구조.</li>
<li><strong>런타임 예측 불가능성</strong>: Effect 내에서의 동기 <code>setState</code> 및 불순한 <code>Date.now()</code> 사용.</li>
<li><strong>테스트 자동화 결핍</strong>: 핵심 로직 변경 시 수동 검증에 의존.</li>
</ol>
<h2>2. 아키텍처 개선: 품질 게이트 도입</h2>
<p>기존의 CI 파이프라인이 단순히 "빌드 가능 여부"만 체크했다면, 개선된 아키텍처는 <strong>3단계 검증 게이트</strong>를 거칩니다.</p>
<h3>품질 검증 파이프라인 아키텍처</h3>
<ol>
<li><strong>Static Analysis (Lint)</strong>: 코드 스타일 및 잠재적 버그 사전 차단.</li>
<li><strong>Unit Tests (Vitest)</strong>: 비즈니스 로직의 회귀 방지.</li>
<li><strong>Build Verification</strong>: 최종 번들링 및 배포 가능성 확인.</li>
</ol>
<h2>3. 실전 코드 예시</h2>
<h3>3.1 린트 에러 해결 (React Purity 확보)</h3>
<p><code>setState</code>를 Effect 내부에서 동기적으로 호출하던 패턴을 클릭 핸들러 내부로 이동시켜 불필요한 렌더링을 방지했습니다.</p>
<pre><code class="language-tsx">// 개선 전
useEffect(() => {
  if (isPlaying) {
    setStartTime(Date.now()); // 불순한 값 의존 및 렌더링 루프 위험
  }
}, [isPlaying]);

// 개선 후
const handleStart = () => {
  setIsPlaying(true);
  const now = Date.now();
  setStartTime(now); // 이벤트 핸들러 내에서 명확한 상태 전이
};
</code></pre>
<h3>3.2 Vitest를 이용한 유틸리티 테스트</h3>
<p><code>src/lib/games.ts</code>와 같은 핵심 데이터 레이어에 대한 단위 테스트를 구축했습니다.</p>
<pre><code class="language-typescript">// src/lib/games.test.ts
import { describe, it, expect, vi } from 'vitest';
import { getGameByDate } from './games';
import fs from 'fs';

vi.mock('fs');

describe('getGameByDate', () => {
  it('존재하지 않는 날짜 요청 시 null을 반환해야 함', async () => {
    (fs.readFileSync as any).mockImplementation(() => {
      throw new Error('File not found');
    });
    
    const result = await getGameByDate('2099-12-31');
    expect(result).toBeNull();
  });

  it('올바른 JSON 파일을 읽어 객체로 파싱해야 함', async () => {
    const mockData = JSON.stringify({ title: 'Test Quiz' });
    (fs.readFileSync as any).mockReturnValue(mockData);
    
    const result = await getGameByDate('2026-03-09');
    expect(result).toEqual({ title: 'Test Quiz' });
  });
});
</code></pre>
<h2>4. 결론: 자동화된 신뢰</h2>
<p>기술 부채 상환의 핵심은 단순히 코드를 고치는 것이 아니라, **"앞으로 나쁜 코드가 들어오지 못하게 막는 시스템"**을 만드는 것입니다. GitHub Actions에 <code>pnpm lint</code>와 <code>pnpm test</code>를 강제함으로써, 이제 이 프로젝트는 최소한의 품질이 보장된 코드만 메인 브랜치에 반영될 수 있는 기반을 갖추게 되었습니다.</p>
