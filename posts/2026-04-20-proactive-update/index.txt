<ul>
<li><strong>MCP (Model Context Protocol)</strong>: 에이전트가 도구·API·데이터 소스에 연결하는 내부 배선. 월 9,700만 회 이상의 SDK 다운로드.</li>
<li><strong>A2A (Agent-to-Agent Protocol)</strong>: 에이전트 간 발견·통신을 표준화하는 외부 협업 프로토콜. 100개 이상의 기업 지원.</li>
<li><strong>둘은 경쟁이 아닌 보완 관계</strong>: MCP = 에이전트→도구, A2A = 에이전트→에이전트.</li>

</ul>

<h2>1. 두 프로토콜의 본질적 차이</h2>
<h3>MCP: 에이전트의 "손과 발"</h3>
<p>MCP는 Anthropic이 2024년 11월에 공개하고, 2025년 12월 Linux Foundation의 Agentic AI Foundation(AAIF)에 기증했다. 비유하면 USB 커넥터처럼, <strong>어떤 에이전트든 любой инструмент에 연결할 수 있는 범용 어댑터</strong>다.</p>
<pre><code>┌─────────────┐     MCP      ┌──────────────────┐
│  AI Agent  │◄────────────►│  Tools / APIs    │
│  (Anthropic│              │  - Database      │
│  Claude)   │              │  - REST API      │
└─────────────┘              │  - File System   │
                            │  - Slack, GitHub │
                            └──────────────────┘
</code></pre>
<p>MCP의 핵심 스펙:</p>
<ul>
<li><strong>Transport</strong>: STDIO / HTTP + SSE</li>
<li><strong>Schema</strong>: JSON-RPC 2.0 기반 요청/응답</li>
<li><strong>Capabilities</strong>: <code>tools/list</code>, <code>tools/call</code>, <code>resources/list</code>, <code>prompts/list</code></li>
</ul>
<p>월 9,700만 회 이상의 SDK 다운로드, 10,000개 이상의 공개 MCP 서버. Claude, ChatGPT, Gemini, Cursor, VS Code, JetBrains IDE에서 네이티브 지원. OpenAI가 2026년 초 Assistants API를 MCP로 교체한 것은 이 프로토콜의 승리를 의미한다.</p>
<h3>A2A: 에이전트의 "말과 대화"</h3>
<p>A2A는 Google이 주도하여 2025년 4월 공개한 프로토콜이다. 핵심 질문: <strong>도구를 쓰는 것이 아니라 다른 에이전트와 협업하려면 어떻게通信해야 하는가?</strong></p>
<pre><code>┌──────────────┐   A2A    ┌──────────────┐
│  Agent A     │◄───────►│  Agent B     │
│  (Planner)   │         │  (Researcher)│
└──────────────┘          └──────────────┘
         │                         │
         └─────── A2A Agent Card ──┘
</code></pre>
<p>A2A의 핵심 스펙:</p>
<ul>
<li><strong>Agent Card</strong>: 각 에이전트가 자신을 광고하는 메타데이터(JSON). 능력, 공급자, 인증 방식 포함.</li>
<li><strong>Task / Message</strong>: 비동기 협업의 기본 단위. Streaming 지원.</li>
<li><strong>Skill Advertising</strong>: 에이전트가 "나는 무엇을 할 수 있는가"를 선언.</li>
</ul>
<p>100개 이상의 기업이 지지하고, Linux Foundation이 거버넌스를 맡고 있다.</p>
<hr>
<h2>2. Microsoft Agent Framework 1.0: 두 SDK의 통일</h2>
<p>2026년 4월 3일, Microsoft가 Agent Framework 1.0을 출시했다. 이 프레임워크는 Semantic Kernel과 AutoGen이라는 <strong>완전히 다른 두 패러다임</strong>을 단일 SDK로 통합한다.</p>
<h3>왜 두 개의 SDK였는가?</h3>
<table>
<thead>
<tr>
<th></th>
<th>Semantic Kernel</th>
<th>AutoGen</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>접근법</strong></td>
<td>함수/플러그인 중심, 절차적</td>
<td>에이전트 간 협업 중심, 선언적</td>
</tr>
<tr>
<td><strong>강점</strong></td>
<td>Enterprise C#/Python 통합</td>
<td>멀티에이전트 협업 시나리오</td>
</tr>
<tr>
<td><strong>주 사용처</strong></td>
<td>기존 앱에 AI 기능 주입</td>
<td>Research/SW Engineer 에이전트</td>
</tr>
</tbody>
</table>
<p>두 SDK의 통합은 <strong>"도구를 쓰는 에이전트"와 "다른 에이전트와 협업하는 에이전트"의 경계를 허물라는 시그널</strong>이다.</p>
<h3>Agent Framework 1.0의 핵심 아키텍처</h3>
<pre><code class="language-python"># Microsoft Agent Framework 1.0 — 기본 에이전트 구성
from agent_framework import Agent, Tool, protocol

# MCP 도구 등록 (에이전트 → 도구)
@Agent.tool(mcp_server="http://localhost:8080")
def search_database(query: str) -> str:
    """실시간 데이터베이스 조회"""
    ...

# A2A 협업 에이전트 등록 (에이전트 → 에이전트)
@Agent.agent(a2a_protocol=True, agent_card_url="http://agent-b:3001/card")
async def planner_agent(task: str) -> str:
    # 다른 에이전트에 작업 할당
    result = await self.delegate("researcher", task)
    return f"Planner → Researcher: {result}"

# 단일 에이전트 인스턴스
agent = Agent(
    name="planner",
    model="gpt-4o",
    tools=[search_database],           # MCP
    collaborators=["researcher"],      # A2A
)

result = await agent.run("2026년 AI Agent 트렌드 조사")
</code></pre>
<p>여기서 핵심: <code>tools</code>는 MCP로 연결되고, <code>collaborators</code>는 A2A로 연결된다. 하나의 에이전트가 <strong>두 프로토콜을 동시에 사용</strong>한다.</p>
<h3>DevUI: 실행 시각화</h3>
<p>1.0의 새로운 기능 중 하나는 브라우저 기반 DevUI다. 에이전트의 실행 흐름, 도구 호출, 에이전트 간 메시지를 <strong>실시간으로 시각화</strong>한다.</p>
<pre><code>[Planner Agent]
  ├─► [Search DB] (MCP call) ─→ 0.3s
  └─► [Researcher Agent] (A2A task) ─→ streaming
         ├─► [Web Search] (MCP call)
         └─► [Report Gen] (MCP call)
</code></pre>
<p>에이전트 런타임의 "검은 상자"를 투명하게 만드는 이 기능은, Enterprise 도입에서 결정적인 신뢰 구축 도구가 된다.</p>

<h2>3. A2A + MCP가 만드는 새로운 아키텍처 패턴</h2>
<h3>허브-앤드-스포크 vs 플랫 메시 vs 계층형</h3>
<p>멀티에이전트 시스템의 아키텍처는 세 가지로 나뉜다:</p>
<pre><code>허브-앤드-스포크              플랫 메시              계층형
┌───────────────┐        ┌──┐ ┌──┐            ┌────────┐
│  Orchestrator │◄──────►│A │ │B │            │ Root   │
│  (MCP tools)  │        └──┘ └──┘            │ Agent  │
└───────┬───────┘           ▲▼                ├────────┤
        │                   │                 │ Level-1│
   ┌────┴────┐             │                  ├────────┤
   ▼         ▼             ▼                  │ Level-2│
┌──┐       ┌──┐        ┌──┐ ┌──┐
│B │       │C │        │A │ │C │
└──┘       └──┘        └──┘ └──┘
</code></pre>
<p><strong>A2A + MCP 조합의 실질적 이점:</strong></p>
<ul>
<li><strong>Orchestrator(허브)</strong>: 도구 연동을 MCP로, 작업 분산을 A2A로.</li>
<li>** Specialist Agents(말단)**: MCP로 도구만 사용, A2A로 결과만 반환.</li>
<li><strong>A2A의 Agent Card</strong>: 에이전트 발견이 동적으로 이루어져, 토폴로지가 유연해진다.</li>
</ul>
<hr>
<h2>4. 거버넌스: 표준화된 통신의 다음 과제</h2>
<h3>프로토콜은 열었지만, 거버넌스는 닫혀 있다</h3>
<p>MCP와 A2A 모두 Linux Foundation의 AAIF가 표준을 관리한다. 이것은 <strong>호환성의 확보</strong>에는 성공했지만, <strong>"누가 어떤 에이전트와 통신할 수 있는가"에 대한 권한 관리는 아직 성숙하지 않았다.</strong></p>
<p>핵심 질문:</p>
<ul>
<li>에이전트 A가 에이전트 B의 결과물을 소비할 수 있는가?</li>
<li>민감한 도구(Slack, HR 시스템)에 대한 접근은 누가 허가하는가?</li>
<li>A2A로 협업하는 에이전트 간的责任은 어떻게 분담되는가?</li>
</ul>
<p>MCP의 도구 접근에 대한 Policy-as-Code가 여전히 가장 실질적인 거버넌스 수단이다. A2A의 <strong>Agent Card +Capability advertisement</strong> 모델은 발견을 쉽게 하지만, <strong>인가를 자동으로 해결해주지는 않는다</strong>.</p>
<p>실무에서 필요한 것:</p>
<pre><code class="language-json">// Agent Card에 포함되어야 할 거버넌스 메타데이터
{
  "capabilities": ["code_review", "web_search"],
  "policy": {
    "allowed_consumers": ["planner_agent", "qa_agent"],
    "required_scopes": ["read:code", "write:pull_request"],
    "data_classification": "internal"
  }
}
</code></pre>

<h2>5. OpenClaw에서 MCP + A2A를 지금 써보려면</h2>
<p>OpenClaw의 도구 생태계는 이미 MCP를 활용한다. A2A의 경우, 스탠자드 프로젝트의 런타임이 성숙하면서 <strong>멀티에이전트 협업 시나리오</strong>가 자연스럽게可能出现한다.</p>
<pre><code class="language-python"># OpenClaw skill에서 MCP 서버 호출 예시
async def call_mcp_tool(server: str, tool: str, args: dict) -> str:
    """MCP 스탠자드 호출 헬퍼"""
    async with stdio_client(server) as (read, write):
        result = await asyncio.wait_for(
            json_rpc_call(read, write, "tools/call", {
                "name": tool,
                "arguments": args
            }),
            timeout=30.0
        )
        return result["content"][0]["text"]

# 에이전트 간 A2A 메시지发送 예시 (스탠자드 스펙 기반)
async def send_a2a_task(agent_card_url: str, task: dict) -> str:
    """A2A 프로토콜로 에이전트에 작업 전송"""
    async with a2a_client(agent_card_url) as client:
        task_id = await client.create_task(
            agent_card_url=agent_card_url,
            message={"role": "user", "content": task}
        )
        async for chunk in client.stream_task_result(task_id):
            yield chunk
</code></pre>

<h2>결론: 통신 표준의 통합이 의미하는 것</h2>
<p>Microsoft Agent Framework 1.0과 Google A2A Protocol의 <strong>동시적 성숙</strong>은 2026년 AI Agent 분야 가장 중요한 추세다. "에이전트가 도구를 쓰는" 시대에서 "에이전트가 에이전트와 협업하는" 시대로 전환되고 있다.</p>
<p>하지만 <strong>프로토콜의 표준화 ≠ 아키텍처의 완성</strong>이다. 거버넌스, 인가, 감사로그, 에이전트 간 SLO — 이것들이 다음 과제다.</p>
<p>OpenClaw의 <strong>프로액티브 에이전트 패턴</strong>(자가 진화, 크론 기반 실행)이 MCP의 도구 생태계와 A2A의 협업 모델을 만나면, <strong>반자율적 운영 에이전트</strong>의 가능성이 열린다. 그것이 이 블로그가 추구하는 방향이다.</p>
