2:T50a9,<h2>왜 지금 이 주제를 다시 봐야 하나</h2>
<p>2026년의 Agent 시스템은 더 이상 <code>tool 호출이 가능하다</code>는 사실만으로 차별화되지 않는다. MCP가 tool interface를 정리해 준 뒤부터, 병목은 훨씬 더 운영적인 곳으로 이동했다.</p>
<ul>
<li>long-running task를 어떻게 추적할 것인가</li>
<li>실패한 실행을 어디까지 재시도할 것인가</li>
<li>실행 결과를 어떤 단위로 검증할 것인가</li>
<li>사람이 개입해야 하는 지점을 어떻게 남길 것인가</li>
<li>사고가 났을 때 어느 로그를 보면 원인을 찾을 수 있는가</li>
</ul>
<p>즉, 문제는 <code>연결</code>이 아니라 <code>운영</code>이다.</p>
<p>최근 2026년 4월의 Agent 트렌드를 보면 이 흐름이 더 선명하다. 업계의 관심은 단순한 model upgrade가 아니라 다음으로 이동하고 있다.</p>
<ul>
<li>runtime security</li>
<li>async task orchestration</li>
<li>execution trace</li>
<li>verification protocol</li>
<li>local or browser based execution isolation</li>
</ul>
<p>이 글에서는 이 흐름을 한 문장으로 압축한다.</p>
<blockquote>
<p>MCP가 tool access를 표준화했다면, 이제 경쟁력은 observable runtime과 verification contract를 얼마나 잘 설계했느냐에서 나온다.</p>
</blockquote>
<p>이번 글은 그 설계를 실무 관점에서 풀어본다.</p>
<hr>
<h2>1. MCP는 시작점일 뿐이다</h2>
<p>MCP는 아주 중요한 문제를 해결했다. model이 어떤 tool을 어떤 schema로 호출해야 하는지 통일된 방식으로 설명할 수 있게 만들었다. 이건 분명히 큰 진전이다.</p>
<p>하지만 실제 운영에서는 MCP만으로 해결되지 않는 질문이 훨씬 많다.</p>
<pre><code class="language-text">사용자 요청
   │
   ▼
Planner model
   │  tool call 생성
   ▼
MCP client
   │
   ▼
MCP server
   │
   ├─ tool A 실행
   ├─ tool B 실행
   └─ tool C 실행
</code></pre>
<p>이 다이어그램은 연결 관점에서는 충분하다. 하지만 운영 관점에서는 빠진 게 많다.</p>
<pre><code class="language-text">사용자 요청
   │
   ▼
Planner
   │
   ▼
Execution Contract Layer
   │  - idempotency key
   │  - timeout budget
   │  - retry policy
   │  - approval policy
   ▼
Runtime Orchestrator
   │  - queue
   │  - state store
   │  - cancellation
   │  - backpressure
   ▼
MCP Tool Runtime
   │
   ├─ Trace Collector
   ├─ Verification Gate
   ├─ Audit Log
   └─ Human Review Hook
</code></pre>
<p>실무에서 차이가 나는 부분은 이 두 번째 그림이다.</p>
<p>MCP는 tool을 부르는 규칙을 제공한다. 하지만 다음은 여전히 여러분이 설계해야 한다.</p>
<ul>
<li>실행을 큐에 넣을지 즉시 돌릴지</li>
<li>retry 가능한 오류와 아닌 오류를 어떻게 나눌지</li>
<li>partial success를 어떤 형태로 저장할지</li>
<li>side effect가 있는 tool call을 재실행해도 되는지</li>
<li>agent가 내린 결론을 바로 반영할지 verification 뒤에 반영할지</li>
</ul>
<p>정리하면 MCP는 <code>interface standard</code>이고, production 경쟁력은 <code>execution standard</code>에서 나온다.</p>
<hr>
<h2>2. 2026년 Agent 시스템의 핵심은 async execution contract다</h2>
<p>많은 팀이 아직도 Agent 실행을 request-response로만 생각한다. 사용자가 요청하면 model이 몇 번 생각하고 tool을 부르고 바로 답을 주는 흐름이다. 작은 데모에는 잘 맞는다. 하지만 실제 업무 자동화는 그렇지 않다.</p>
<p>예를 들어 이런 작업을 생각해 보자.</p>
<ul>
<li>GitHub issue를 읽고 재현 환경을 만든다</li>
<li>staging에 배포해서 smoke test를 돈다</li>
<li>browser로 로그인과 결제 흐름을 확인한다</li>
<li>결과를 정리해서 PR comment를 남긴다</li>
</ul>
<p>이건 몇 초 안에 끝나는 작업이 아니다. 중간에 network jitter도 있고, external API limit도 있고, 사람이 승인해야 하는 단계도 있다. 이 시점부터 필요한 것은 model intelligence보다 <code>async execution contract</code>다.</p>
<h3>실행 계약이 필요한 이유</h3>
<p>실행 계약이 없으면 이런 문제가 터진다.</p>
<ul>
<li>같은 작업이 중복 실행된다</li>
<li>timeout 뒤에 실제 실행은 계속된다</li>
<li>성공과 실패가 섞인 상태가 저장되지 않는다</li>
<li>사용자는 실패했다고 보는데 외부 시스템은 이미 변경되었다</li>
<li>review 없이 side effect가 발생한다</li>
</ul>
<p>이를 막으려면 모든 agent task가 최소한 아래 필드를 가져야 한다.</p>
<pre><code class="language-ts">export type TaskStatus =
  | 'queued'
  | 'running'
  | 'waiting_approval'
  | 'waiting_external'
  | 'verifying'
  | 'completed'
  | 'failed'
  | 'cancelled';

export interface AgentTask {
  taskId: string;
  requestId: string;
  idempotencyKey: string;
  status: TaskStatus;
  createdAt: string;
  updatedAt: string;
  budget: {
    deadlineMs: number;
    maxRetries: number;
    maxToolCalls: number;
  };
  input: {
    userGoal: string;
    riskLevel: 'low' | 'medium' | 'high';
  };
  output?: {
    summary: string;
    artifacts: string[];
  };
  error?: {
    code: string;
    message: string;
    retryable: boolean;
  };
}
</code></pre>
<p>핵심은 <code>agent가 생각한 내용</code>보다 <code>실행 단위가 어떤 상태로 흘러가는가</code>를 먼저 데이터 모델로 고정하는 것이다.</p>
<h3>실행 계약의 최소 규칙</h3>
<p>내가 추천하는 최소 규칙은 다음 다섯 가지다.</p>
<ol>
<li>모든 side effect 작업은 <code>idempotencyKey</code>를 가진다.</li>
<li>모든 task는 <code>deadline</code>과 <code>retry budget</code>를 가진다.</li>
<li>approval이 필요한 단계는 별도 상태로 빠진다.</li>
<li>tool result는 원문과 요약을 함께 저장한다.</li>
<li>completed 전에 반드시 <code>verifying</code> 단계를 통과한다.</li>
</ol>
<p>이 규칙만 지켜도 운영 난이도가 크게 내려간다.</p>
<hr>
<h2>3. Observable Runtime은 로그가 아니라 실행 그래프다</h2>
<p>많은 팀이 observability를 <code>tool call 로그 남기기</code> 정도로 이해한다. 하지만 Agent에서는 그걸로 부족하다. 왜냐하면 문제의 핵심이 단일 호출 실패가 아니라, 호출들의 연쇄에서 발생하기 때문이다.</p>
<p>예를 들어 이런 상황을 보자.</p>
<ul>
<li>planner가 browser tool을 호출했다</li>
<li>login step은 성공했다</li>
<li>checkout page에서 AB test variation이 달라졌다</li>
<li>fallback selector가 실패했다</li>
<li>agent는 empty result를 요약하면서 <code>상품이 없다</code>고 결론 냈다</li>
</ul>
<p>이 사건을 사후 분석하려면 단순 로그가 아니라 실행 그래프가 필요하다.</p>
<pre><code class="language-text">task-1842
 ├─ step-1 plan.generate                420ms   success
 ├─ step-2 browser.open                 980ms   success
 ├─ step-3 browser.login               2120ms   success
 ├─ step-4 browser.find-cart-button    1300ms   failed(selector_not_found)
 ├─ step-5 fallback.search-button       880ms   failed(selector_not_found)
 ├─ step-6 model.summarize              610ms   success
 └─ verdict                            incorrect
</code></pre>
<p>여기서 중요한 건 세 가지다.</p>
<ul>
<li><strong>causality</strong>: 어떤 실패가 다음 판단에 영향을 줬는가</li>
<li><strong>timing</strong>: 느린 구간이 어디인가</li>
<li><strong>decision trace</strong>: model이 왜 그런 결론을 냈는가</li>
</ul>
<h3>어떤 이벤트를 남겨야 하나</h3>
<p>실무에서 최소한 아래 이벤트는 구조화해서 남겨야 한다.</p>
<pre><code class="language-ts">interface RuntimeEvent {
  eventId: string;
  taskId: string;
  stepId: string;
  type:
    | 'plan_created'
    | 'tool_started'
    | 'tool_succeeded'
    | 'tool_failed'
    | 'approval_requested'
    | 'approval_granted'
    | 'verification_started'
    | 'verification_failed'
    | 'task_completed';
  ts: string;
  actor: 'planner' | 'tool' | 'verifier' | 'human';
  toolName?: string;
  latencyMs?: number;
  payload: Record&#x3C;string, unknown>;
}
</code></pre>
<p>주의할 점도 있다. Agent observability는 무조건 많이 쌓는다고 좋은 게 아니다. raw prompt와 raw artifact를 전부 적재하면 비용도 커지고 privacy 문제도 생긴다. 그래서 관측 단위를 나눠야 한다.</p>
<ul>
<li>control plane event: 상태 전이, 승인, 취소, 재시도</li>
<li>execution event: tool input hash, output digest, latency, error code</li>
<li>audit artifact: 필요 시만 전문 저장</li>
</ul>
<p>즉, <code>항상 저장할 것</code>과 <code>문제가 생겼을 때만 열람할 것</code>을 분리해야 한다.</p>

<h2>4. Verification은 마지막에 붙이는 QA가 아니다</h2>
<p>많은 팀이 verification을 <code>결과 나온 뒤에 한번 더 검사하는 단계</code>로 붙인다. 그런데 Agent 시스템에서는 그렇게 하면 늦는 경우가 많다. 이미 side effect가 발생했기 때문이다.</p>
<p>Verification은 사후 검사보다 <code>실행 경계</code>로 들어가야 한다.</p>
<h3>어떤 경계에 verifier를 둘 것인가</h3>
<p>가장 효과적인 방식은 아래 세 지점에 verifier를 두는 것이다.</p>
<ol>
<li><strong>plan gate</strong>: 위험한 tool 조합인지 본다.</li>
<li><strong>pre-commit gate</strong>: side effect 반영 직전에 본다.</li>
<li><strong>post-condition gate</strong>: 반영 뒤 시스템 상태가 기대와 맞는지 본다.</li>
</ol>
<p>예를 들어 GitHub issue triage agent라면 이렇게 된다.</p>
<pre><code class="language-text">issue 분석
  ▼
plan gate
  - destructive action 포함 여부
  - 외부 발송 여부
  - budget 초과 여부
  ▼
sandbox 실행
  ▼
pre-commit gate
  - 재현 성공 증거 존재 여부
  - 로그와 요약의 일치 여부
  ▼
comment 작성 또는 label 변경
  ▼
post-condition gate
  - comment 생성 확인
  - label 반영 확인
</code></pre>
<h3>Verifier는 다른 model이어야 하나</h3>
<p>꼭 그렇지는 않다. 더 중요한 건 역할 분리다.</p>
<ul>
<li>planner: 일을 진행하려는 성향</li>
<li>verifier: 틀린 결론을 막으려는 성향</li>
</ul>
<p>같은 model을 쓰더라도 prompt contract와 입력 자료를 다르게 구성하면 역할 분리는 가능하다. 다만 high-risk workflow라면 planner와 verifier를 다른 model 혹은 다른 policy budget으로 분리하는 편이 낫다.</p>
<p>아래는 간단한 verification gate 예시다.</p>
<pre><code class="language-ts">interface VerificationInput {
  taskId: string;
  intendedAction: string;
  evidence: Array&#x3C;{
    kind: 'log' | 'screenshot' | 'json' | 'diff';
    uri: string;
    digest: string;
  }>;
  assertions: string[];
}

interface VerificationResult {
  passed: boolean;
  score: number;
  reasons: string[];
  requiresHumanReview: boolean;
}

export async function verifyBeforeCommit(
  input: VerificationInput,
): Promise&#x3C;VerificationResult> {
  const missingEvidence = input.assertions.filter((assertion) => {
    return !input.evidence.some((item) => item.kind === 'json' || item.kind === 'diff');
  });

  if (missingEvidence.length > 0) {
    return {
      passed: false,
      score: 0.42,
      reasons: [`핵심 근거 부족: ${missingEvidence.join(', ')}`],
      requiresHumanReview: true,
    };
  }

  return {
    passed: true,
    score: 0.91,
    reasons: ['필수 근거 확인 완료'],
    requiresHumanReview: false,
  };
}
</code></pre>
<p>이 코드는 단순해 보이지만 중요한 원칙을 담고 있다.</p>
<ul>
<li>verifier는 <code>생각이 그럴듯한가</code>가 아니라 <code>근거가 충분한가</code>를 본다.</li>
<li>verification 실패는 곧바로 human review로 연결된다.</li>
<li>score는 편의 기능일 뿐, 최종 판정은 rules와 evidence completeness가 담당한다.</li>
</ul>
<hr>
<h2>5. Planner와 Runtime을 분리하지 않으면 생기는 문제</h2>
<p>Agent를 처음 만들 때 흔히 하는 실수는 planner가 runtime 정책까지 동시에 쥐게 만드는 것이다. 예를 들어 model이 직접 이런 걸 결정하게 두는 식이다.</p>
<ul>
<li>몇 번 재시도할지</li>
<li>timeout을 얼마나 줄지</li>
<li>approval 없이 진행할지</li>
<li>background task로 넘길지</li>
</ul>
<p>이 구조는 데모에서는 빨라 보인다. 하지만 운영에서는 재앙에 가깝다. 왜냐하면 실행 정책은 business policy와 security policy의 영역이지, 매번 model이 새로 추론할 영역이 아니기 때문이다.</p>
<p>권장 구조는 다음과 같다.</p>
<pre><code class="language-text">Planner
  - 어떤 목표를 달성할지 계획
  - 어떤 tool이 필요한지 제안

Policy Engine
  - 이 계획이 허용 범위 안인지 판단
  - retry, timeout, approval 규칙 부여

Runtime Orchestrator
  - 실제 큐잉, 취소, 상태 저장, 재실행 수행

Verifier
  - 실행 근거와 결과의 적합성 검사
</code></pre>
<p>아래 예시는 planner의 자유도를 줄이고 runtime이 정책을 강제하는 형태다.</p>
<pre><code class="language-ts">type ToolCategory = 'read' | 'write' | 'external_send' | 'deploy';

interface PlannedToolCall {
  toolName: string;
  category: ToolCategory;
  params: Record&#x3C;string, unknown>;
}

interface ExecutionPolicy {
  timeoutMs: number;
  retryLimit: number;
  requiresApproval: boolean;
  queue: 'inline' | 'background';
}

export function resolvePolicy(call: PlannedToolCall): ExecutionPolicy {
  switch (call.category) {
    case 'read':
      return {
        timeoutMs: 10_000,
        retryLimit: 1,
        requiresApproval: false,
        queue: 'inline',
      };
    case 'write':
      return {
        timeoutMs: 20_000,
        retryLimit: 0,
        requiresApproval: true,
        queue: 'inline',
      };
    case 'external_send':
      return {
        timeoutMs: 15_000,
        retryLimit: 0,
        requiresApproval: true,
        queue: 'background',
      };
    case 'deploy':
      return {
        timeoutMs: 120_000,
        retryLimit: 0,
        requiresApproval: true,
        queue: 'background',
      };
  }
}
</code></pre>
<p>이 구조의 장점은 분명하다.</p>
<ul>
<li>planner가 똑똑해도 policy를 우회할 수 없다.</li>
<li>runtime metrics를 category 기준으로 바로 집계할 수 있다.</li>
<li>review 기준이 코드로 남는다.</li>
<li>운영자가 model behavior보다 policy file을 먼저 볼 수 있다.</li>
</ul>
<p>실제로 production 운영은 <code>model tuning</code>보다 <code>policy clarity</code>에서 더 자주 개선된다.</p>
<hr>
<h2>6. Browser runtime과 local runtime을 같은 방식으로 다루면 안 된다</h2>
<p>2026년 Agent 흐름에서 browser runtime의 비중은 계속 커지고 있다. 하지만 browser tool을 local file tool과 같은 재시도 규칙으로 다루면 장애가 늘어난다. 실패 특성이 다르기 때문이다.</p>
<ul>
<li>local runtime 실패: permission error, file not found, process exit code</li>
<li>browser runtime 실패: selector drift, auth expiry, timing race, modal interrupt</li>
</ul>
<p>즉, tool abstraction은 같아도 failure model은 다르다.</p>
<p>그래서 runtime layer는 최소한 아래처럼 분리돼야 한다.</p>
<pre><code class="language-text">Local Tool Adapter
  - deterministic failure 비중이 높음
  - 재현 가능성 높음
  - retry보다 즉시 실패가 더 유익할 때 많음

Browser Tool Adapter
  - nondeterministic failure 비중이 높음
  - DOM variation과 timing 영향 큼
  - bounded retry와 fallback selector가 유효함
</code></pre>
<p>실무 팁 하나를 덧붙이면, browser tool에서는 <code>정답 추론</code>보다 <code>증거 수집</code>을 우선시하는 편이 낫다. 예를 들면 다음 순서다.</p>
<ol>
<li>page state snapshot 저장</li>
<li>실패 selector 기록</li>
<li>screenshot 또는 accessibility tree 저장</li>
<li>그 다음에만 fallback reasoning 수행</li>
</ol>
<p>이 순서를 거꾸로 하면, evidence 없는 요약이 늘어난다.</p>
<hr>
<h2>7. Production 체크리스트</h2>
<p>실제 팀이 Agent runtime을 운영하기 전에 반드시 점검해야 할 항목을 짧게 정리한다.</p>
<h3>실행 계약</h3>
<ul>
<li>모든 task에 <code>taskId</code>, <code>requestId</code>, <code>idempotencyKey</code>가 있는가</li>
<li>timeout과 retry budget이 정책으로 분리되어 있는가</li>
<li>cancellation이 실제 tool execution까지 전파되는가</li>
<li>partial failure 상태가 저장되는가</li>
</ul>
<h3>observability</h3>
<ul>
<li>task 단위 상태 전이가 구조화 이벤트로 남는가</li>
<li>tool latency와 error code가 집계되는가</li>
<li>planner decision과 verifier decision이 구분되어 기록되는가</li>
<li>raw artifact 저장 정책과 digest 저장 정책이 나뉘어 있는가</li>
</ul>
<h3>verification</h3>
<ul>
<li>side effect 직전 gate가 존재하는가</li>
<li>evidence 부족 시 human review로 빠지는가</li>
<li>verifier score만으로 commit하지 않는가</li>
<li>post-condition check가 실제 external state를 확인하는가</li>
</ul>
<h3>security</h3>
<ul>
<li>tool category별 최소 권한 정책이 있는가</li>
<li>external send, deploy, delete는 무조건 approval 대상인가</li>
<li>audit log가 수정 불가능한 저장소로 복제되는가</li>
<li>prompt와 artifact에 민감 정보 마스킹이 적용되는가</li>
</ul>
<h3>운영</h3>
<ul>
<li>stuck task를 감지하는 watchdog가 있는가</li>
<li>queue backlog가 일정 임계치를 넘으면 backpressure가 작동하는가</li>
<li>model outage 시 degraded mode가 존재하는가</li>
<li>workflow별 SLO를 정의했는가</li>
</ul>
<hr>
<h2>8. 어떤 지표를 봐야 운영이 좋아지는가</h2>
<p>Agent 시스템에서 자주 빠지는 함정이 <code>정확도</code> 하나만 보는 것이다. 하지만 운영 지표는 더 입체적이어야 한다.</p>
<p>내가 추천하는 기본 지표는 아래와 같다.</p>
<ul>
<li><strong>Task Success Rate</strong>: 최종 목표 달성 비율</li>
<li><strong>Verification Pass Rate</strong>: verifier를 통과한 비율</li>
<li><strong>Time to Useful Action</strong>: 첫 유효 작업까지 걸린 시간</li>
<li><strong>Human Intervention Rate</strong>: 사람 개입 비율</li>
<li><strong>Replay Safety Rate</strong>: 같은 task 재실행 시 side effect 없이 복구되는 비율</li>
<li><strong>Evidence Completeness Score</strong>: 판단에 필요한 근거 충족 비율</li>
</ul>
<p>이 중 특히 중요한 건 <code>Verification Pass Rate</code>와 <code>Evidence Completeness Score</code>다. 왜냐하면 Agent 시스템의 실패는 종종 <code>틀린 답</code>보다 <code>근거 없는 자신감</code>으로 나타나기 때문이다.</p>
<hr>
<h2>결론: 다음 경쟁력은 더 영리한 model이 아니라 더 설명 가능한 실행이다</h2>
<p>2026년의 Agent 시스템은 분명 더 강력해졌다. 하지만 그만큼 더 위험해졌다. 이유는 단순하다. 이제 Agent는 읽고 요약하는 수준을 넘어, 실제로 실행하고 바꾸고 보낸다.</p>
<p>그래서 다음 단계의 핵심 질문은 이것이다.</p>
<ul>
<li>이 agent는 무엇을 실행했는가</li>
<li>왜 그렇게 실행했는가</li>
<li>어떤 근거로 통과되었는가</li>
<li>실패했다면 어느 단계에서 어긋났는가</li>
<li>다시 실행해도 안전한가</li>
</ul>
<p>이 질문에 답할 수 없으면, tool이 백 개 있어도 production 신뢰도는 올라가지 않는다.</p>
<p>MCP는 매우 중요한 기반이다. 하지만 실무 팀이 진짜로 만들어야 하는 것은 그 위의 운영 계층이다.</p>
<ul>
<li>async execution contract</li>
<li>structured runtime events</li>
<li>verification gate</li>
<li>policy enforced orchestration</li>
<li>evidence first debugging</li>
</ul>
