10:T565b,<h2>TL;DR</h2>
<ul>
<li>코딩 에이전트가 실패하는 주요 원인은 LLM 추론이 아니라 <strong>도구 실행 파이프라인(tool harness)의 불완전함</strong>이다.</li>
<li><strong>Agent Harness Engineering</strong>은 에이전트의 도구 호출을 설계, 실행, 감사, 복구하는 포괄적 엔지니어링 분야로 emergence하고 있다.</li>
<li>이 분야의 핵심 문제 4가지: Tool Interface 신뢰성, Sandbox 격리, Execution Budget 관리, Failure Recovery</li>
<li>OpenClaw, Claude Code, Codex 등 주요 에이전트 런타임의 harness 설계 비교 분석</li>
<li>전통 DevOps 엔지니어에게 이 분야는 AI-native 인프라 구축의 가장 접근하기 쉬운 진입점이다.</li>
</ul>
<hr>
<h2>1. 패러다임의 전환: "추론"에서 "실행"으로</h2>
<p>2023~2025년, 코딩 에이전트 생태계는 이른바 **"Model Wars"**로 요약되었다. GPT-4o vs Claude Sonnet vs Gemini 2.5 — 어떤 모델이 코딩 태스크에서 더優秀인가를 논하는 것이 생태계의 중심话题이었다.</p>
<p>2026년 중반, 이 풍경이 근본적으로 변하고 있다. 모델 성능이 정체점에 접근하면서, 에이전트 개발자들의 관심사가 <strong>추론 계층</strong>에서 <strong>실행 계층</strong>으로 이동하고 있다. 구체적으로:</p>
<blockquote>
<p><strong>"우리 모델은 충분하다. 문제는 도구를 어떻게 믿을 수 있느냐다."</strong></p>
</blockquote>
<p>이 전환의 근거로 세 가지 신호를 든다:</p>
<p><strong>1. Tool Call Failure Rate의 가시화</strong>
Claude Code, Codex, Cursor 내부 데이터를 보면, 실제로 코드 변경에 실패하는 케이스 중 <strong>60~70%가 LLM 추론이 아닌 도구 실행 단계</strong>에서 발생한다. 파일 시스템 권한 오류, 잘못된 Git 상태, 비동기 명령의 Race Condition 등이 대표적이다.</p>
<p><strong>2. Harness-oriented 프로젝트의 급등</strong>
<code>awesome-agent-harness</code> 컬렉션이 2026년 1분기에 3배 성장했고, <code>oh-my-pi</code>, <code>agent-orchestrator</code>, <code>composiohq/agent-orchestrator</code> 같은 프로젝트들이 "프레임워크가 아니라 harness를 만들어라"는 메시지로 빠르게 스타를 모으고 있다.</p>
<p><strong>3. DevOps 영역과의 Convergence</strong>
Harness Engineering은 전통적 DevOps가 해온 작업 — 빌드/테스트 파이프라인, 격리된 실행 환경, 장애 복구 — 을 에이전트 영역으로 확장한 것이다. SRE의 Playbook이 에이전트의 Self-Healing 파이프라인이 되는 구조.</p>
<hr>
<h2>2. Agent Harness의 구조적 해부</h2>
<h3>2-1. Harness란 무엇인가</h3>
<p>Agent Harness는 에이전트가 외부 세계와 상호작용하는 <strong>실행 파이프라인</strong> 전체를 가리킨다. 구체적으로 네 개의 하위 시스템으로 구성된다:</p>
<pre><code>┌─────────────────────────────────────────────────────────┐
│                    AGENT HARNESS                         │
├──────────────┬──────────────┬───────────────┬────────────┤
│ Tool Registry │  Execution   │  Result       │  Failure   │
│ &#x26; Discovery   │  Sandbox     │  Parser       │  Recovery  │
│              │  Manager      │  &#x26; Validator  │  Engine    │
└──────────────┴──────────────┴───────────────┴────────────┘
</code></pre>
<p><strong>Tool Registry</strong>: 에이전트가 호출 가능한 도구를 등록, 검색, version 관리한다.
<strong>Execution Sandbox</strong>: 도구 실행을 격리된 환경에서 수행하여 호스트 시스템을 보호한다.
<strong>Result Parser</strong>: 도구 실행 결과를 파싱하여 에이전트의 다음 행동을 결정한다.
<strong>Failure Recovery</strong>: 실행 실패 시 재시도, 부분 롤백, 대안 경로 선택을 담당한다.</p>

<p>에이전트에게 "파일을 읽어라"는 명령은 간단해 보이지만, 실제로는 다음과 같은 분기가 존재한다:</p>

<h2>3. Execution Sandbox: 격리가 곧 신뢰성이다</h2>
<h3>3-1. Sandbox 설계의 세 가지 접근법</h3>
<p>도구 실행의 격리 수준에 따라 세 가지 아키텍처로 나뉜다:</p>
<p><strong>Level 1 — Process Isolation (가장 흔함)</strong></p>
<pre><code>에이전트 → tool_call → subprocess.Popen() → 격리 프로세스 실행 → stdout 수집
</code></pre>
<ul>
<li>장점: 구현 단순, 모든 OS에서 동작</li>
<li>단점: 컨테이너보다 느림, 리소스 격리 제한</li>
</ul>
<p><strong>Level 2 — Container Isolation (Container Agents)</strong></p>
<pre><code>에이전트 → tool_call → container.run(image, cmd) → 결과 반환
</code></pre>
<ul>
<li>장점: 완전한 환경 격리, 재현 가능한 실행 환경</li>
<li>단점: Cold Start 지연, 이미지 관리 오버헤드</li>
</ul>
<p><strong>Level 3 — WASM Micro-VM (가장 엄격함)</strong></p>
<pre><code>에이전트 → tool_call → WASM runtime.execute(module) → 격리된 사이드_effect만 허용
</code></pre>
<ul>
<li>장점: 컨테이너보다 100배 빠른 시작, 네이티브에 가까운 성능</li>
<li>단점: 호환되는 도구 스펙trum이 제한적, 복잡한 시스템콜 에뮬레이션 필요</li>
</ul>
<h3>3-2. MEnvAgent: 10개 프로그래밍 언어의 테스트 환경을 자동 구축하는 멀티에이전트</h3>
<p>MEnvAgent(Multi-Environment Agent)는 2026년 주목할 만한 연구로, 에이전트에게 <strong>검증 가능한 실행 환경</strong>을 자동으로 구축하는 프레임워크다.</p>
<p>핵심 구조:</p>
<pre><code class="language-python"># MEnvAgent 아키텍처 (Pseudo-code)
class MEnvAgent:
    def __init__(self, language: str, task: str):
        self.env_builder = EnvBuilderAgent()      # 환경 구축 담당
        self.executor = ExecutionAgent()          # 테스트 실행 담당
        self.verifier = VerificationAgent()       # 결과 검증 담당

    async def run(self) -> ExecutionResult:
        # 1단계:Planning — 필요한 환경 파악
        plan = await self.planner.decompose(f"""
            "{task}"를 수행하려면 {self.language} 환경이 필요합니다.
            의존성: X, Y, Z. 테스트 스크립트: A, B, C.
        """)

        # 2단계:Execution-Verification Loop
        for attempt in range(3):
            env = await self.env_builder.build(plan, language=self.language)
            result = await self.executor.run(task, env=env)
            verified = await self.verifier.check(result, expected=plan.assertions)

            if verified.success:
                return result
            else:
                # 검증 실패 → 환경을 재구축하고 재시도
                plan = await self.planner.revise(plan, failure=verified.failure)

        raise MaxAttemptsExceeded()
</code></pre>
<p><strong>핵심 통찰</strong>: 에이전트에게 "코드를 작성하라"고 시키는 것만으로는 불충분하다. <strong>실행 가능한 환경이 먼저 제공되어야</strong> 에이전트의 다음 판단이 유효하다. 이것이 MEnvAgent가 제시하는 Paradigm Shift다.</p>
<h3>3-3. oh-my-pi: Terminal 기반 코딩 에이전트의 Harness 설계</h3>
<p><code>oh-my-pi</code>는 최근 GitHub에서 급상승한 terminal 기반 코딩 에이전트로, harness 설계에서 주목할 점이 있다:</p>
<pre><code class="language-bash"># omp commit — deterministic pipeline 실행
omp commit --push --dry-run --no-changelog --context ./my-project

# 핵심: hash-anchored edits
# 특정 파일+라인 조합을 해시로 고정하여, 컨텍스트 윈도우가 작아져도 edit의 대상을 정확히 파악
</code></pre>
<pre><code class="language-yaml"># .omp/config.yaml — harness 설정 예시
harness:
  execution:
    runtime: tmux          # Persistent session으로 cold start 제거
    sandbox_level: process  # Process isolation (container보다 빠름)
    allowed_syscalls:
      - read
      - write
      - exec
      - getdents  # 파일 시스템 목록 조회
    blocked_syscalls:
      - mount
      - syslog
      - reboot

  tool_retry:
    max_attempts: 3
    backoff: exponential
    retryable_errors:
      - EACCES      # 권한 일시적 문제
      - ENOENT     # 파일이 나중에 생성될 수 있음
      - ETIMEDOUT  # 네트워크 타임아웃

  result_validation:
    schema_strict: true      # ToolResult 스키마 강제
    allowed_return_types:
      - string
      - number
      - boolean
      - object
    max_output_bytes: 10485760  # 10MB 제한
</code></pre>
<p><strong>oh-my-pi의 핵심 혁신</strong>: Persistent IPython Kernel 기반 Python 코드 실행. 에이전트가 Python 코드 실행 결과를 IPython 커널 상태에서 재사용하므로, 매번 새로운 Python 프로세스를 띄우는 오버헤드를 제거한다. 이는 에이전트가 중간 계산 결과를 재사용해야 하는 시나리오에서 결정적이다.</p>

<h2>4. Failure Recovery: 에이전트의 Self-Healing 파이프라인</h2>
<h3>4-1. 네 가지 실패 모드와 대응 전략</h3>
<p>코딩 에이전트의 실패는 크게 네 가지로 분류된다:</p>
<table>
<thead>
<tr>
<th>실패 모드</th>
<th>원인</th>
<th>회복 전략</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>Tool Execution Failure</strong></td>
<td>파일 권한, 네트워크, 경로 오류</td>
<td>Retry + Fallback Tool</td>
</tr>

<h3>4-2. Execution Budget: 에이전트가 자원을 낭비하지 않는 구조</h3>
<p>Agentic Workflow에서 가장 위험한 상황 중 하나는 에이전트가 <strong>무한 루프</strong>에 빠지는 것이다. 동일한 실패를 반복하며 토큰을 소진하는 상황.</p>
<p>이를 방지하는 <strong>Execution Budget</strong> 패턴:</p>
<pre><code class="language-typescript">interface ExecutionBudget {
  maxToolCalls: number;          // 최대 도구 호출 횟수
  maxTotalTokens: number;        // 최대 토큰 소비량
  maxWallTimeMs: number;         // 최대 실제 소요 시간
  costBudgetUSD: number;         // 최대 비용
}

class BudgetEnforcer {
  enforce(budget: ExecutionBudget, context: AgentContext): void {
    const remaining = this.calculateRemaining(context);
    if (remaining.toolCalls &#x3C;= 0) {
      throw new BudgetExceededError("Maximum tool calls reached");
    }
    if (remaining.tokens &#x3C;= 0) {
      throw new BudgetExceededError("Context window exhausted");
    }
    // 다음 도구 호출 시 budget 정보를 주입
    context.hints.push({
      budget: remaining,
      strategy: this.suggestStrategy(remaining)
    });
  }

  suggestStrategy(remaining: Remaining): Strategy {
    if (remaining.wallTimeMs &#x3C; 30_000) {
      return "prefer_cache_and_short_calls";
    }
    if (remaining.tokens &#x3C; 50_000) {
      return "aggressive_pruning";
    }
    return "normal";
  }
}
</code></pre>
<h3>4-3. agent-orchestrator: CI 자동 수정 파이프라인</h3>
<p>ComposioHQ의 <code>agent-orchestrator</code>는 실제 CI 파이프라인에 에이전트를 integrate한 케이스다:</p>
<pre><code class="language-yaml"># agent-orchestrator.yaml
defaults:
  runtime: tmux
  agent: claude-code
  workspace: worktree

projects:
  my-app:
    repo: owner/my-app
    path: ~/my-app

reactions:
  ci-failed:
    auto: true
    action: send-to-agent
    retries: 2           # 실패 시 2회 자동 재시도
    escalationAfter: 30m  # 30분 내 해결 안 되면 인간에게 알림

  changes-requested:
    auto: true
    action: send-to-agent  # 리뷰어 코멘트 → 에이전트가 자동 반영
    escalateAfter: 30m
</code></pre>
<p>이 설정의 핵심은 <strong>auto: true</strong>가 에이전트에게 완전한 자율성을 부여하되, <strong>escalationAfter</strong>로 인간이 항상 개입할 수 있는 안전장치를 둔다는 점이다. 에이전트에게 "너 혼자 해결해"라고 하면서도, 시간 제한을 둔다.</p>

<h2>5. Harness Engineering의 4가지 핵심 원칙</h2>
<h3>원칙 1: Tool 계약은 명시적으로, 기본값은 안전하게</h3>
<pre><code class="language-typescript">// 모든 도구는 항상 ToolResult를 반환해야 함
type ToolHandler&#x3C;TInput, TOutput> = (
  input: TInput,
  context: ToolContext
) => Promise&#x3C;ToolResult&#x3C;TOutput>>;

// 기본값: 실패 가능한 동작은 명시적 거부
const defaultToolConfig = {
  timeoutMs: 30_000,
  maxOutputBytes: 1024 * 1024,  // 1MB
  retryable: false,              // 기본적으로 재시도 안 함
  sandboxed: true               // 기본적으로 샌드박스 실행
};
</code></pre>
<h3>원칙 2: 실패는 복구 가능하게 설계하라</h3>
<pre><code class="language-python">#失败了 → 복구 불가능한가? 아니라면 설계 문제
class ToolExecution:
    def execute(self, tool, args):
        try:
            return tool.execute(args)
        except RetryableError as e:
            # 지数적 백오프 후 재시도
            for attempt in range(self.max_retries):
                time.sleep(2 ** attempt)
                try:
                    return tool.execute(args)
                except RetryableError:
                    continue
            # 그래도 실패 → 부분 롤백 시도
            return self.partial_rollback(tool, args)
        except FatalError as e:
            # 복구 불가능 → 실패 로그 + 인간 알림
            self.notify_human(f"Tool {tool.name} failed fatally: {e}")
            raise
</code></pre>
<h3>원칙 3: 실행 환경은 reproducible해야 한다</h3>
<pre><code class="language-bash"># 불변的环境 — 같은 입력에 항상 같은 결과
docker build -t agent-sandbox:python-3.12 -f Dockerfile.sandbox .

# Harness는 도구 실행 결과를 버전 관리한다
git add .harness/snapshots/v3/
git commit -m "harness: capture execution state for session 4f8a9b"
</code></pre>

<h2>6. 전통 DevOps 엔지니어에게 이 분야가 중요한 이유</h2>
<h3>6-1. 진입 장벽이 낮다</h3>
<p>DevOps 엔지니어는 이미 harness와同类한 것을 구축하고 있다:</p>
<ul>
<li><strong>CI/CD 파이프라인</strong> = 에이전트의 실행 파이프라인</li>
<li><strong>Container/VM 격리</strong> = 에이전트의 Sandbox</li>
<li><strong>Retry/Backoff 정책</strong> = 에이전트의 Failure Recovery</li>
<li><strong>Monitoring/Alerts</strong> = 에이전트의 Observability</li>
</ul>
<p>다른 점은 <strong>대상工件</strong>이 코드가 아니라 LLM 프롬프트/도구 호출이라는 것뿐이다.</p>
<h3>6-2. 수요가 폭발적이다</h3>
<p><code>awesome-harness-engineering</code>_repo의 설명에서:</p>
<blockquote>
<p>Demonstrates harness design for scientific workflows where individual turns can exceed model context limits but the overall pipeline must maintain coherence across days.</p>
</blockquote>
<p>수일에서 수주간 동작하는 장기 실행 에이전트 파이프라인을 설계하고 운영하는 일은, 현재 생태계에서 가장人手不足인 영역이다.</p>
<h3>6-3. 구체적인 학습 경로</h3>
<pre><code>Step 1: OpenClaw 또는 Claude Code의 harness 소스 코드 읽기
  → 실제 production harness가 어떻게 동작하는지 확인

Step 2: MCP 서버를 직접 만들어보기
  → 도구 인터페이스 설계의 실제 문제 파악

Step 3: 에이전트 실행 로그 분석하기
  → 실패 패턴을 분류하고, 그에 맞는 Recovery 전략 설계

Step 4: 자신만의 harness 프레임워크 만들기
  → 재사용 가능한 추상화 레이어 구축
</code></pre>

<h2>7. 결론: Harness가 곧 에이전트의 경쟁력이다</h2>
<p>2026년, 코딩 에이전트 생태계는 <strong>"추론 모델"的时代에서 "실행 환경"의 시대</strong>로 진입하고 있다. 같은 GPT-4o를 사용하더라도, Tool Registry 설계가 잘못되면 에이전트는频繁한 실패를 반복한다.</p>
<p><strong>Agent Harness Engineering</strong>은 이 문제를 엔지니어링적으로 해결하는 분야다. 도구 인터페이스의 신뢰성, 격리된 실행 환경, 예산 기반 실행 관리, 체계적 실패 복구 — 이 네 가지 요소가 모두 갖춰져야 에이전트가 프로덕션 환경에서 안정적으로 동작한다.</p>
<p>주인님의 OpenClaw는 이미 이 방향으로 설계되어 있다. MCP 런타임, skill 시스템, exec sandbox — 이것들이 전부 harness의 구성 요소다. 이를 의식적으로 설계하고 개선하는 것이 다음 단계의 핵심 과제다.</p>
<p><strong>실행 가능한 다음 액션:</strong></p>
<ol>
<li><code>awesome-agent-harness</code> GitHub repo를 탐독하여 최신 harness 패턴 학습</li>
<li>OpenClaw의 현재 tool 실행 파이프라인을 audit하고, Retry 정책과 Budget 제한 검토</li>
<li>자신만의 MCP 서버를 하나 만들어보며 Tool Interface 설계의 실제 문제 경험</li>
</ol>
