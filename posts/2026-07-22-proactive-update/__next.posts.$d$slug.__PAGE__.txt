<h2>2. Gateway 성능 모델의 3대 차원</h2>
<p>ASTP Gateway의 성능은 세 가지 축에서 정의된다:</p>
<h3>2.1 Throughput (RPS)</h3>
<p>Stateless MCP 환경에서 Gateway의 Throughput은 <strong>WAL Write Latency</strong>에 의해 제한된다. 각 요청은 Gateway에 도착하면 WAL에 기록되고, CRDT Delta로 변환된 후 DRT(Delta Routing Table)에 따라 peer Gateway로 전파된다.</p>
<pre><code>Request → [WAL Append] → [CRDT Delta Serialize] → [DRT Lookup] → [Peer Send]
           t_wal          t_serialize                    t_send
</code></pre>
<p>총 처리 시간: <code>T_total = max(t_wal, t_serialize + t_send)</code></p>
<p>여기서 <code>t_wal</code>이 지배적이다. WAL은 fsync 기반의 내구성 보장이 필요하므로 (PIPA 감사 증적 요구사항), disk I/O가 병목이 된다.</p>
<h3>2.2 Latency (P50/P95/P99)</h3>
<p>Latency는 다음 구성 요소로 분해된다:</p>
<pre><code>L_total = L_network(tcp) + L_tls + L_request_parse + L_wal + L_serialize + L_drt + L_send
</code></pre>
<p>Stateless MCP에서 <code>L_tls</code>는 connection reuse로 무시 가능한 수준이지만, 첫 요청의 connection setup 비용은 여전히 존재한다. ASTP에서는 HTTP/2 multiplexing과 connection warm-up pool로 이 비용을 분산한다.</p>
<p><strong>핵심 발견:</strong> Stateless 전환으로 가장 큰 Latency 개선이 일어나는 구간은 <code>L_drt</code>가 아니다 (ASTP는 이미 분산 처리). 오히려 <strong>sticky session 제거로 인한 load imbalance 해소</strong>가 P99 Latency를 40% 이상 개선한다.</p>
<h3>2.3 Consistency (DRT Sync Delay)</h3>
<p>ASTP의 Consistency는 DRT(Delta Routing Table)의 Sync Delay로 측정된다. DRT는 각 Gateway가 peer로부터 받은 CRDT Delta를 적용하는 Clock으로, 이 delay가 클수록 <strong>Stale Read</strong> 가능성이 높아진다.</p>
<p><code>Sync_Delay(t) = max(Clock_i(t) - Clock_j(t)) for all i, j in Gateway Mesh</code></p>
<p>벤치마크에서 측정할 핵심 메트릭은 <strong>Sync_Delay의 분포</strong>다. 강한 일관성을 요구하는 PIPA 감사 증적 시나리오에서는 Sync_Delay &#x3C; 50ms를 SLO로 설정한다.</p>

<h2>4. Capacity Planning 모델: Little's Law + M/G/1 Queue</h2>
<h3>4.1 Gateway를 Queue로 모델링하기</h3>
<p>ASTP Gateway는 Stateless MCP 위에서 동작하므로, 각 Gateway 인스턴스를 <strong>M/G/1 Queue</strong> (Poisson arrival, General service time, 1 server)로 모델링할 수 있다.</p>
<p><strong>Little's Law:</strong> <code>L = λ × W</code></p>
<ul>
<li><code>L</code>: In-flight requests (대기 중인 요청 수)</li>
<li><code>λ</code>: Arrival rate (RPS)</li>
<li><code>W</code>: Average time in system (latency)</li>
</ul>
<p><strong>M/G/1 Pollaczek-Khinchine 공식:</strong></p>
<pre><code>W = (λ × E[S²]) / (2 × (1 - ρ)) + E[S]

where:
  ρ = λ × E[S] (utilization)
  E[S] = 평균 service time
  E[S²] = service time의 2차 모멘트
</code></pre>
<p>이 공식은 Gateway의 <strong>서비스 시간 분산(variance)이 Latency에 얼마나 치명적인지</strong> 정량화한다. CRDT Delta 직렬화 크기가 들쭉날쭉할수록 (한국어 텍스트가 포함된 Delta vs 짧은 JSON Delta), <code>E[S²]</code>가 커지고 대기열 Latency가 폭발한다.</p>
<h3>4.2 실전 Capacity Equation</h3>
<pre><code>RequiredNodes = ceil(λ_target / λ_single_node × replication_factor)

where:
  λ_single_node = min(1/E[S] × CPU_limit, IOPS_limit / IOPS_per_request)
  replication_factor = 3 (ASTP Gateway Mesh 기본값)
</code></pre>
<p>실전에서는 CPU 한계와 IOPS 한계 중 낮은 쪽이 결정적 요소가 된다.</p>
<p><strong>WAL이 병목일 때:</strong></p>
<pre><code class="language-typescript">function calculateWALCapacity(config: WALConfig): CapacityEstimate {
  const fsyncLatency = config.isNVMe ? 0.01 : 0.5; // ms (NVMe vs SSD)
  const batchSize = config.fsyncBatchSize; // records per fsync
  const maxWritesPerCore = 1000 / (fsyncLatency / batchSize); // writes/sec/core

  const throughputPerCore = maxWritesPerCore * config.avgRecordSize;
  const ioBandwidth = config.diskWriteBandwidth; // MB/s

  return {
    maxRPS: Math.floor(maxWritesPerCore * config.cores),
    ioBound: (throughputPerCore * config.cores) > ioBandwidth,
    bottleneck: (throughputPerCore * config.cores) > ioBandwidth
      ? 'disk_bandwidth'
      : 'wal_fsync_latency'
  };
}
</code></pre>

<h3>4.3 PIPA 보관 요구사항과 Capacity 충돌</h3>
<p>PIPA 감사 증적 요구사항은 <strong>모든 WAL Record를 1년간 보관</strong>해야 한다. 이는 단순한 저장소 문제를 넘어서 <strong>WAL Compaction 정책</strong>이 Capacity Planning의 주요 변수가 된다.</p>
<pre><code>WAL 저장량 = λ × avg_record_size × 365일 × replication_factor
            = 50,000 × 2KB × 31,536,000초 / 3
            ≈ 1.05 TB/year (single Gateway 기준)
            ≈ 3.15 TB/year (3-Gateway Mesh)

NVMe SSD 3.84TB 기준 → 1년 미만으로 보관 불가
→ WAL Tiered Storage (Hot WAL: NVMe 7일 → Cold Archive: S3/object storage 358일)
</code></pre>
<p>이 Tiered WAL 전환은 Auto-Benchmark에서 <strong>Hot-to-Cold 전환 시점의 Latency Spike</strong>를 측정해야 하는 근거가 된다.</p>

<h3>5.2 Split-brain Detection 성능 변화</h3>
<p>Stateless 환경에서 Split-brain Detection은 <strong>기존보다 더 중요해진다</strong>. Session-based에서는 각 Gateway가 "이 요청은 내가 처리 중"이라는 implicit ownership을 가졌지만, stateless에서는 요청이 여러 Gateway에 분산될 수 있기 때문이다.</p>
<p>ASTP의 Split-brain Detection은 DRT의 Clock 단조성 검사로 동작한다:</p>
<pre><code class="language-typescript">interface SplitBrainDetector {
  // Clock_i(t) - Clock_j(t) > threshold → split-brain
  checkClockMonotonicity(gatewayId: string, clock: VectorClock): boolean {
    for (const [peerId, peerClock] of this.drt.clocks.entries()) {
      const deviation = Math.abs(clock[peerId] - peerClock[peerId]);
      if (deviation > SPLIT_BRAIN_THRESHOLD) {
        return false; // split-brain detected
      }
    }
    return true;
  }
}
</code></pre>
<p>벤치마크에서 측정된 <strong>Split-brain Detection Latency:</strong></p>
<table>

<h3>5.3 CRDT Delta Size와 네트워크 대역폭</h3>
<p>Stateless MCP에서 각 요청은 독립적인 CRDT Delta를 생성한다. Delta Size는 <strong>요청 Payload 크기 + Gateway 메타데이터</strong>로 결정되며, 한국어 Payload가 포함될 경우 UTF-8 인코딩으로 인해 3배까지 커질 수 있다.</p>
<pre><code>Delta = { gatewayId, timestamp, clock, crdtType, operation, payload }

Payload size impact:
  English:  "approve_transaction_12345"      → 28 bytes
  Korean:   "거래승인_12345"                   → 18 bytes (UTF-8, 더 compact)
  Mixed:    "transaction_거래승인_12345"        → 27 bytes
</code></pre>
<p>흥미롭게도 한국어 Payload가 오히려 <strong>더 작은 경우가 많다.</strong> 숫자+영문+한글 혼용 시에도 한글은 UTF-8에서 3바이트지만, 영문 숫자 조합보다 실질적 payload에서 더 효율적인 경우가 있다.</p>

<h2>8. 자가 검토 (Self-Critique)</h2>
<p>이 글에서 의도적으로 다루지 않거나 한계를 인정하는 부분:</p>
<ol>
<li>
<p><strong>실험실 벤치마크와 실전 환경의 차이:</strong> Section 7의 예상 결과는 통제된 환경(L4 switch + dedicated NVMe + 동일 리전) 기준이다. 실제 클라우드 환경(공유 tenancy, cross-region latency, egress cost)에서는 결과가 크게 달라질 수 있다.</p>
</li>
<li>
<p><strong>M/G/1 모델의 한계:</strong> Gateway의 Service Time은 CRDT Delta Merge에서 발생하는 <strong>Lock Contention</strong> 때문에 strictly independent가 아니다. Heavy tail 상황에서는 M/G/1 예측이 20~30% 과소평가될 가능성이 있다. G/G/1이나 Simulation-based 접근이 더 정확할 수 있다.</p>
</li>
<li>
<p><strong>한국어 Delta Size 편향:</strong> Section 5.3에서 언급했듯이, 필수 메타데이터(clock vector, gatewayId 등)가 Delta의 지배적 요소이므로 한국어 Payload가 전체 Delta Size에 미치는 영향은 생각보다 크지 않다. UTF-8 우려는 Benchmark Driver에 언어별 Payload Generator를 포함하지 않으면 검증할 수 없다.</p>
</li>
<li>
<p><strong>Split-brain Detection의 False Positive Threshold:</strong> Section 5.2의 Threshold(SYNC_DEVIATION = 50ms)는 3-Gateway Mesh 기준이다. 7-Gateway 이상으로 확장하면 network chaos로 인한 transient deviation이 200ms를 넘는 경우가 발생할 수 있어, <strong>Dynamic Threshold Calibration</strong>이 필요하다. Auto-Benchmark가 이 Dynamic Calibration까지 자동화해야 실전에서 유용하다.</p>
</li>
<li>
<p><strong>WAL Tiered Storage 전환의 Atomicity:</strong> Section 4.3에서 Hot→Cold 전환을 언급했지만, 이 과정에서의 <strong>Read-Write Conflict</strong>를 다루지 않았다. Cold로 이동 중인 WAL Block에 대한 Read 요청이 들어오면 어떻게 처리할 것인가? Shadow Copy, Copy-on-Write, 또는 Gentle Stop-the-world Compaction 중 선택이 필요하다.</p>
</li>
<li>
<p><strong>PIPA 5년 보관 vs GDPR 잊힐 권리 (Right to be Forgotten) 충돌:</strong> 의료 Federation 시나리오에서 PIPA 5년 보관은 1년 보관보다 엄격하지만, GDPR "잊힐 권리" 청구가 들어오면 어떻게 할 것인가? Merkle Hash 기반의 <strong>Incremental Expunge</strong> (WAL에서 특정 record 제거 + 증적 유지)가 필요하지만, 이는 WAL의 append-only 속성과 충돌한다. ZKP로 "해당 record는 존재하지 않는다"를 증명할 수 있지만, 이는 #069 Observability에서도 언급한 한계다.</p>
</li>
<li>
<p><strong>Auto-Benchmark가 Gateway 자체에 미치는 부하:</strong> Section 3의 Benchmark Framework는 성능 측정을 위해 Gateway에 추가 부하를 가한다. 특히 Online Monitor Mode에서 Spike 감지 시 Auto-Benchmark를 Trigger하면, 이미 Spiking 중인 Gateway에 추가 부하가 가해져서 <strong>Observability Paradox</strong> (관찰 행위가 시스템을 변화시킴)가 발생한다. 이 문제는 <strong>Out-of-band Tracing</strong>과 <strong>Adaptive Sampling</strong>으로 완화해야 한다.</p>
</li>
<li>
<p><strong>재무 모델 부재:</strong> Capacity Planning의 출력은 "Gateway 몇 대"였지만, 실제 의사결정자는 <strong>비용 모델</strong>(EC2/GKE 노드 비용, NVMe vs EBS gp3 비용, Cross-region Data Transfer 비용)을 함께 봐야 한다. 벤치마크 결과 → 비용 예측 → CAPEX/OPEX 승인으로 이어지는 파이프라인이 실전 Capacity Planning의 완전체다.</p>
</li>
</ol>
