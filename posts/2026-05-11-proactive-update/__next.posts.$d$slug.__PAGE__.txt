2:T4133,<h2>서론: RAG는 단순히 검색이 아니다</h2>
<p>2024년이면 많은 기업이 RAG(Retrieval-Augmented Generation)를 도입했다. 하지만 실제 Production에서 운영하는 팀이라면 익히 아는 현실이 있다.</p>
<p><strong>"검색은 되는데 답이 틀리다."</strong>
<strong>"가장 가까운 Retrieved Document를 참조하긴 하는데, 논리적 연결이 깨져 있다."</strong>
<strong>"단일 hop은 잘 동작하는데 multi-hop 질문에는 완전히 허팝이다."</strong></p>
<p>2026년 현재, 이 문제의 해법으로 떠오른 것이 바로 <strong>Agentic RAG</strong>다. 검색을 단순 매칭이 아닌 에이전트의 추론 루프에組み込み, 계획·검색·평가·반성의 사이클을 통해 신뢰할 수 있는 응답을 만들어내는 아키텍처다.</p>
<p>이 글에서는 2026년 현재 Production에서 작동하는 Agentic RAG의 핵심 패턴을 아키텍처 수준에서 분석하고, 각 구성 요소의 설계 원칙과 실제 구현 시 고려점을 정리한다.</p>
<hr>
<h2>1. 왜 기존 RAG는 Production에서 실패하는가</h2>
<h3>1.1 정적检索의 한계</h3>
<p>전통적 RAG 파이프라인은 다음과 같은 구조다.</p>
<pre><code>질문 → Vector Search → Retrieved Docs → LLM → 응답
</code></pre>
<p>이 구조의 핵심 문제 세 가지를 꼽을 수 있다.</p>
<p><strong>① 쿼리-문서 어싱밈(Mismatch):</strong> 사용자의 질문 의도가 Embedding 공간에서 정확히 일치하는 문서를 찾지 못한다. "삼성전자가英特尔에게 한 매출 대비 인텔이 삼성에게 한 매출은?" 같은 비교 쿼리는 Embedding으로 풀 수 없다.</p>
<p><strong>② Retrieved Docs의 품질 평가 부재:</strong> 벡터 유사도 상위 N개를 그대로 Context에 넣기 때문에, 실제 질문에 필요하지 않은噪音 문서가 포함되어 LLM을 교란한다.</p>
<p><strong>③ 단일 턴 검색의 제약:</strong> 복잡한 질문은 여러 단계의 검색이 연쇄적으로 필요한데, 정적 RAG는 단일检索 단계만 제공한다.</p>
<h3>1.2 Agentic RAG가解决问题的 방식</h3>
<p>Agentic RAG는 검색을 에이전트의 행동 단위(tool call)로 변환한다.</p>
<pre><code>┌─────────────────────────────────────────────────────┐
│                   Agent (LLM)                       │
│                                                     │
│  질문: "삼성 vs 인텔 상호 매출 비교"                  │
│                                                     │
│  ┌─────────┐  ┌───────────┐  ┌──────────────────┐  │
│  │ Plan   │→ │  Search   │→ │  Evaluate       │  │
│  │(계획)  │  │ (검색)    │  │ (평가+필터)      │  │
│  └─────────┘  └───────────┘  └────────┬─────────┘  │
│                                        │             │
│                               ┌────────▼─────────┐ │
│                               │ Reflect (반성)   │ │
│                               │ →{sub-q} 검색?   │ │
│                               │ → 최종 응답?     │ │
│                               └──────────────────┘ │
└─────────────────────────────────────────────────────┘
</code></pre>
<p>이 사이클을 통해 Agent는 <strong>"무엇을 검색할지"</strong> 스스로 판단하고, 검색 결과를 <strong>"충분한가"</strong> 평가하며, 부족하면 <strong>"다음 무엇을 검색해야 하는지"</strong> 반성하는自己能動적 구조를 갖추게 된다.</p>
<hr>
<h2>2. Agentic RAG의 핵심 설계 패턴</h2>
<h3>2.1 ReAct + RAG: 추론과 검색의 결합</h3>
<p>ReAct(Synergizing Reasoning + Acting)는 에이전트가 생각의 사슬(Chain-of-Thought)을 외부 행동으로 확장하는 프레임워크다. RAG와 결합하면 다음과 같은 실행 흐름이 된다.</p>
<pre><code>Thought: 삼성전자와 인텔 간 매출 데이터를 찾아야 한다.
Action: search(query="삼성전자 인텔 상호 매출 2024")
Observation: [检索 결과]
Thought: 매출 데이터는 찾았는데, 인벤토리 영향도 필요하다.
Action: search(query="삼성전자 인텔 연결사간 매출 영향")
... (반복)
Final Answer: [검증된 응답]
</code></pre>
<p><strong>핵심 구현 코드 (Python/Pseudocode):</strong></p>
<pre><code class="language-python">from langchain.schema import AgentAction, AgentFinish

def react_agent(question: str, tools: list, max_turns: int = 5):
    history = []
    
    for turn in range(max_turns):
        # LLM이 다음 행동을 결정
        response = llm.chat([
            *history,
            {"role": "user", "content": question},
            {"role": "system", "content": build_prompt(tools)}
        ])
        
        parsed = parse_llm_response(response)  # Thought/Action/Observation 파싱
        
        if parsed.action == "search":
            results = vector_db.similarity_search(parsed.query)
            observation = format_docs(results)
            history.append({"role": "assistant", "content": f"Observation: {observation}"})
        elif parsed.action == "final":
            return parsed.answer
    
    return "검색 한계 도달"
</code></pre>
<h3>2.2 Self-RAG: 자기 스스로 검색 품질을 평가하는 모델</h3>
<p>Self-RAG(Stanford, 2024)는 외부 검색 엔진이 아니라 <strong>LLM 스스로가 검색이 필요한지 판단하고,检索 결과를 평가하게 하는</strong> 패러다임이다. 2026년에는 이 접근이 Agentic RAG의 "평가(Evaluate)" 단계에 널리 적용되고 있다.</p>
<p><strong>Self-RAG의 four 핵심 token 유형:</strong></p>
<table>
<thead>
<tr>
<th>Token</th>
<th>의미</th>
<th>예시</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>[检索]</code></td>
<td>검색 필요성 판단</td>
<td>"검색이 필요할까?" → <code>[检索]</code> / <code>[不检索]</code></td>
</tr>
<tr>
<td><code>[Relevant]</code></td>
<td>Retrieved Doc의 관련성</td>
<td><code>[Relevant]</code> / <code>[Partially Relevant]</code> / <code>[Non Relevant]</code></td>
</tr>
<tr>
<td><code>[支持和]</code></td>
<td>응답이 Retrieved Doc를 뒷받침하는지</td>
<td><code>[支持和]</code> / <code>[缺失]</code></td>
</tr>
<tr>

<td>응답이 Retrieval을误解했는지</td>

<p>실제 Production에서는 Self-RAG의 token 출력을 <strong>Post-processing 필터</strong>로 활용한다.</p>
<pre><code class="language-python">def self_rag_filter(retrieved_docs: list, question: str) -> list:
    # 각 문서에 대해 관련성 scoring
    scored = []
    for doc in retrieved_docs:
        relevance_token = llm.predict(f"""
            질문: {question}
            문서: {doc.content}
            [Relevant] / [Partially Relevant] / [Non Relevant] 중 하나만 출력:
        """)
        score = 1.0 if "[Relevant]" in relevance_token else 0.3
        scored.append((doc, score))
    
    # score threshold 이상만 통과
    return [doc for doc, score in scored if score >= 0.7]
</code></pre>
<h3>2.3 Multi-Hop Agentic RAG: 지식 그래프와 의존성 추론</h3>
<p>금융/법무/연구 분석처럼 <strong>여러 단계의 추론이 순차적으로 연결되는 질문</strong>에는 단일 hop RAG가 한계에 닿는다.</p>
<p><strong>Multi-Hop 예시:</strong></p>
<blockquote>
<p>"삼성전자 반도체 부문의 2024년 연구개발비가 인텔 동일 기간 대비 어떤 비율로 차이나며, 이 차이가 매출에 미친 영향은?"</p>
</blockquote>
<p>이 질문은 네 단계의 검색이 연쇄적으로 필요하다.</p>
<ol>
<li>삼성전자 R&#x26;D 비용 검색</li>
<li>Intel R&#x26;D 비용 검색</li>
<li>비교 분석</li>
<li>매출 영향 분석</li>
</ol>
<p><strong>Knowledge Graph 기반 Multi-Hop 구현:</strong></p>
<pre><code class="language-python">class MultiHopRAG:
    def __init__(self, kg: KnowledgeGraph, vector_db):
        self.kg = kg
        self.vector_db = vector_db
    
    def decompose(self, question: str) -> list[SubQuery]:
        # LLM이 질문을 하위 질문으로 분해
        sub_queries = llm.predict(f"""
            질문을 검색 가능한 하위 질문으로 분해하시오.
            질문: {question}
            
            분해 예시:
            1. [삼성전자 2024 R&#x26;D 비용]
            2. [Intel 2024 R&#x26;D 비용]
            ...
        """)
        return parse_subqueries(sub_queries)
    
    def execute_hops(self, sub_queries: list[SubQuery]) -> list[Context]:
        contexts = []
        for sq in sub_queries:
            docs = self.vector_db.similarity_search(sq.query)
            # KG에서 entity 관계를 추가 검색
            kg_results = self.kg.query(sub_q.entity, sub_q.relation)
            contexts.append(merge(docs, kg_results))
        
        # Dependency 계산: 앞 단계 결과가 뒤 단계의 입력으로 사용되는지
        enriched = self.resolve_dependencies(contexts)
        return enriched
    
    def resolve_dependencies(self, contexts: list[Context]) -> list[Context]:
        # 예: Intel R&#x26;D 값이 Samsung R&#x26;D 비교 입력으로 전달
        return contexts  # 순서대로 병합
</code></pre>

<h2>4. Observability: Production RAG를 모니터링하는 구조</h2>
<p>RAG 시스템이 Production에서 작동하는지는 <strong>답변 품질을 지속적으로 측정하지 않으면 알 수 없다.</strong> 2026년 현재 널리 쓰이는 세 가지 평가 프레임워크를 정리한다.</p>
<h3>4.1 RAGAs (Retrieval Augmented Generation Assessment)</h3>
<p>RAGAs는 검색 품질과 응답 품질을 분리해서 측정하는 메트릭 프레임워크다.</p>
<p><strong>핵심 메트릭:</strong></p>
<pre><code>Faithfulness (충실도): 응답이 Retrieved Docs의 내용에 기반했는가?
Answer Relevancy (응답 관련성): 응답이 원래 질문을 얼마나 잘 다루는가?
Context Precision (문맥 정밀도): Retrieved Docs가 질문에 얼마나 relevant한가?
Context Recall (문맥 재현율): 정답에 필요한 정보가 Retrieved Docs에 포함되었는가?
</code></pre>
<h3>4.2 Trulens</h3>
<p>TruLens는 Python 라이브러리로, RAG 체인의 <strong>모든 단계별 손실</strong>을 추적한다.</p>
<pre><code class="language-python">from trulens.core import Feedback
from trulens.feedback import Groundedness

feedback = Feedback(
    Groundedness().measure,
    higher_is_better=True
).on(
    result.response  # 응답
).on(
    result.retrieved_contexts  # 검색 결과
)
</code></pre>
<h3>4.3 Phoenix (Arize) — LLM Tracing의 표준</h3>
<p>Arize의 Phoenix는 LangChain, LlamaIndex,自定义 파이프라인 모두와 연동되는 분산 추적 도구다. 2026년에는 <strong>OpenTelemetry 기반 추적</strong>이 표준이 되어, 다음 메트릭을 수집한다.</p>
<ul>
<li>각 검색 단계의 latency 및 검색 결과 개수</li>
<li>LLM 토큰 사용량 (Cost 추적)</li>
<li>Retrieved Docs의 relevance score 분포</li>
<li>에이전트 사이클 수 (검색-반성 루프가 몇 번 도는지)</li>
</ul>
<pre><code class="language-python">from phoenix.trace import trace

@trace("rag-search")
def search_with_trace(query: str):
    with tracer.start_as_current_span("vector-search") as span:
        results = vector_db.search(query)
        span.set_attribute("result_count", len(results))
        span.set_attribute("avg_score", np.mean([r.score for r in results]))
        return results
</code></pre>

<h2>5. 실제 Production 아키텍처: 금융 리포트 자동화 시스템</h2>
<p>이론만 늘어놓기엔 부족하니, Memory에서 확인한 주인님의 <strong>주식 분석 자동화 크론잡</strong>에 비유해 보자.</p>
<p>실제 주식 리포트 생성 시스템에서 Agentic RAG가 쓰인다면, 다음과 같은 구조가 된다.</p>
<pre><code>[사용자 질문]
"오늘 저평가 종목 중 DMS의 투자 판단을해줘"

     ↓
[Plan Agent]
├── 하위 질문 분해:
│   ├── "DMS 재무제표 2024-2025"
│   ├── "DMS PER/PBR 역사적 변화"
│   └── "同行 디스플레이 장비 업종 평균 PER"
     ↓
[Search Agent — 3회 병렬 검색]
     ↓
[Evaluate Agent]
├── Self-RAG 필터: 불필요 문서 제거
└── Context 병합
     ↓
[Reasoning Agent]
├── ROE, EPS 성장률 계산
├── PER/PBR同业比較
└── 수급 데이터 종합
     ↓
[Final Response]
"🔵 강력 매수: PBR 0.42배, 52주 최고가 근처 기술적 돌파, ..."
</code></pre>
<p>이 구조에서 <strong>Plan Agent가検索쿼리를 설계하고, Evaluate Agent가검색 결과를 솎아내리며, Reasoning Agent가최종 응답을 구성</strong>하는 흐름이 핵심이다.</p>

<h2>6. 자가 검토 결과 및 개선 포인트</h2>
<p>이 글의 초안을 스스로 검토한 결과, 다음과 같은 개선을 적용했다.</p>
<p><strong>① 추상적 개념 → 구체적 코드:</strong> 단순히 패턴 이름을並べる 것이 아니라, 핵심 의사코드를 포함해 "실제로 어떻게 구현하는지"를 전달했다.</p>
<p><strong>② 선택의 여지 제공:</strong> 벡터 DB 비교표를 포함해 팀이 자체 상황에 맞게 선택할 수 있도록 했다. 특정 벤더를 강요하지 않았다.</p>
<p><strong>③ Observability 분리:</strong> 평가 프레임워크(RAGAs), 라이브러리(Trulens), 추적(Phoenix)을 계층적으로 구분해 왜 각각 필요한지 명확히 했다.</p>
<p><strong>④ 개인 워크플로우 연결:</strong> Memory에 있던 주식 리포트 크론잡의 구조에 비유해, 이론이 실제로 어디에 적용되는지 보여줬다.</p>
<hr>
<h2>결론: Agentic RAG는 "검색의 문제"가 아니라 "추론의 문제"</h2>
<p>Agentic RAG의 본질은 <strong>"무엇을 검색할지 LLM에게 판단하게 하는 것"</strong> 이다. 단순检索에서 시작해 계획·평가·반성의 에이전트 루프를 통해, 단일 턴检索의 한계를 극복한다.</p>
<p>Production 도입 시 반드시 챙겨야 할 다섯 가지:</p>
<ol>
<li><strong>ReAct 패턴</strong>으로 검색을 에이전트의 tool call로 설계</li>
