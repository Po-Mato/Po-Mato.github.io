10:T210d,<h2>배경: 왜 지금 local-first와 zero-server가 다시 뜨는가</h2>
<p>오늘 기준 GitHub Trending에는 <code>google-ai-edge/gallery</code>, <code>LiteRT-LM</code>, <code>GitNexus</code>, <code>qmd</code> 같은 프로젝트가 강하게 올라와 있습니다. 결은 서로 달라 보여도 공통점은 분명합니다. <strong>추론과 코드 인텔리전스를 최대한 사용자 가까이 끌어오려는 흐름</strong>입니다. 이유는 세 가지입니다.</p>
<p>첫째, latency입니다. agent는 대화 한 번으로 끝나지 않고 계획, 검색, 도구 호출, 검증을 연쇄적으로 수행합니다. 매 단계가 원격 round trip을 타면 체감 속도는 급격히 나빠집니다. 둘째, privacy입니다. 코드베이스, 회의 메모, 운영 문서처럼 민감한 맥락을 로컬에 남기고 싶어 하는 팀이 많아졌습니다. 셋째, cost 구조입니다. 항상 중앙 서버를 거치지 않아도 되는 작업까지 비싼 inference path에 태우는 방식은 오래 못 갑니다.</p>
<p>문제는 여기서 시작됩니다. local-first agent는 빠르고 싸고 사적인 대신, <strong>중앙 로그와 운영 통제가 약해지기 쉽습니다.</strong> Hacker News에서 <code>Project Glasswing</code>, <code>GLM-5.1</code>, agent orchestration testbed 같은 주제가 동시에 주목받는 것도 같은 맥락입니다. 이제 시장은 capability만 보지 않습니다. 긴 실행을 얼마나 안정적으로 끝내는지, 실패를 얼마나 설명 가능하게 남기는지를 봅니다.</p>
<h2>아키텍처: local-first agent stack의 실제 병목</h2>
<p>로컬 실행이 된다고 운영이 쉬워지는 건 아닙니다. 오히려 병목은 더 숨습니다. 대표 구조는 아래와 같습니다.</p>
<pre><code class="language-text">[UI / CLI]
   -> [Planner]
   -> [Local Context Index: files, SQLite, IndexedDB]
   -> [Tool Runtime: shell, browser, MCP, local scripts]
   -> [Verifier]
   -> [Execution Journal + Metrics]
</code></pre>
<p>이 구조에서 핵심은 model이 아니라 runtime입니다. planner가 도구를 잘 골라도 verifier가 없으면 잘못된 결과를 성공으로 착각합니다. tool runtime이 빨라도 journal이 없으면 어떤 단계에서 틀어졌는지 알 수 없습니다. 즉, zero-server agent의 경쟁력은 "로컬에서 돈다"가 아니라 <strong>"로컬에서도 운영이 된다"</strong> 에 있습니다.</p>
<h2>Agent SLO 설계: request metric이 아니라 task metric을 봐야 한다</h2>
<p>기존 웹 서비스처럼 p95 latency와 200 응답률만 보면 반쪽짜리입니다. agent는 의미 실패가 더 많기 때문입니다. 실무에서는 아래 네 가지부터 잡는 편이 좋습니다.</p>
<ul>
<li><code>task_success_rate</code>: 최종 산출물이 검증까지 통과한 비율</li>
<li><code>time_to_useful_action_ms</code>: 첫 유의미한 초안, 패치, 검색 결과를 만들기까지의 시간</li>
<li><code>verification_pass_rate</code>: verifier가 결과를 승인한 비율</li>
<li><code>recovery_rate</code>: 중간 실패 뒤 fallback 또는 재시도로 복구한 비율</li>
</ul>
<p>추천 기준도 request가 아니라 run 단위여야 합니다.</p>
<pre><code class="language-json">{
  "service": "local-agent-runtime",
  "window": "7d",
  "slo": {
    "task_success_rate": 0.92,
    "p95_time_to_useful_action_ms": 30000,
    "verification_pass_rate": 0.95,
    "recovery_rate": 0.60
  }
}
</code></pre>
<p>이 지표들이 중요한 이유는 간단합니다. local-first stack은 인프라 장애보다 <strong>도구 선택 오류, 컨텍스트 누락, 검증 실패</strong>가 더 자주 문제를 만듭니다. 그래서 runtime이 얼마나 똑똑했는지보다, 실패를 얼마나 빨리 감지하고 회복했는지가 더 큰 차이를 만듭니다.</p>
<h2>Execution Journal 스키마: 블랙박스를 운영 데이터로 바꾸기</h2>
<p>Execution Journal은 에이전트의 step 로그를 단순 텍스트가 아니라 집계 가능한 구조로 남기는 방식입니다. 최소 스키마는 아래 정도면 충분합니다.</p>
<pre><code class="language-json">{
  "runId": "run_2026_04_08_001",
  "spanId": "plan.s2.tool.browser",
  "step": 2,
  "toolCall": {
    "name": "browser.snapshot",
    "args": { "target": "host", "url": "https://github.com/trending" }
  },
  "latencyMs": 1840,
  "verification": {
    "status": "pass",
    "rule": "contains_project_names"
  },
  "recoveryHint": "snapshot 실패 시 web_fetch로 축소 수집 후 sourceConfidence를 낮춰 기록"
}
</code></pre>
<p>포인트는 다섯 가지입니다.</p>
<ul>
<li><code>spanId</code>: trace와 journal을 연결하는 축</li>
<li><code>step</code>: planner 순서를 복원하는 최소 단서</li>
<li><code>toolCall</code>: 실패를 재현할 수 있는 실행 문맥</li>
<li><code>latencyMs</code>: 체감 성능과 병목 분석의 기준</li>
<li><code>verification</code>과 <code>recoveryHint</code>: 단순 기록이 아니라 다음 행동을 유도하는 운영 데이터</li>
</ul>
<h2>TypeScript 예시: journal을 남기며 도구 실행하기</h2>
<p>아래 예시는 Next.js 또는 Node.js runtime 어디서든 쓸 수 있는 단순 패턴입니다.</p>
<pre><code class="language-ts">import { appendFile } from "node:fs/promises";

type JournalEntry = {
  runId: string;
  spanId: string;
  step: number;
  toolCall?: { name: string; args: Record&#x3C;string, unknown> };
  latencyMs: number;
  verification: { status: "pass" | "fail"; rule: string };
  recoveryHint?: string;
};

async function record(entry: JournalEntry) {
  await appendFile("./var/execution-journal.jsonl", JSON.stringify(entry) + "\n");
}

export async function runToolWithJournal&#x3C;T>(input: {
  runId: string;
  spanId: string;
  step: number;
  toolName: string;
  args: Record&#x3C;string, unknown>;
  invoke: () => Promise&#x3C;T>;
  verify: (result: T) => boolean;
}) {
  const started = Date.now();

  try {
    const result = await input.invoke();
    const passed = input.verify(result);

    await record({
      runId: input.runId,
      spanId: input.spanId,
      step: input.step,
      toolCall: { name: input.toolName, args: input.args },
      latencyMs: Date.now() - started,
      verification: {
        status: passed ? "pass" : "fail",
        rule: "result-schema-and-domain-check"
      },
      recoveryHint: passed ? undefined : "fallback to cached context or ask human review"
    });

    if (!passed) throw new Error("verification_failed");
    return result;
  } catch (error) {
    await record({
      runId: input.runId,
      spanId: input.spanId,
      step: input.step,
      toolCall: { name: input.toolName, args: input.args },
      latencyMs: Date.now() - started,
      verification: { status: "fail", rule: "runtime-exception" },
      recoveryHint: "retry once, then degrade to read-only path"
    });
    throw error;
  }
}
</code></pre>
<p>이 패턴의 장점은 화려하지 않지만 확실합니다. 나중에 Prometheus나 OpenTelemetry를 붙이더라도, 먼저 JSONL 기반 journal만으로 run 단위 회고와 실패 분석이 가능해집니다.</p>
<h2>운영 체크리스트: local agent를 실제 서비스로 만들려면</h2>
<ul>
<li>쓰기 동작 전에는 항상 verifier를 두기</li>
<li>외부 부작용이 있는 step은 approval gate와 분리하기</li>
<li>tool 실패를 숨기지 말고 <code>recoveryHint</code>까지 남기기</li>
<li>cache hit, fallback 사용 여부를 journal에 같이 기록하기</li>
<li>성공률보다 <code>time_to_useful_action</code> 악화를 먼저 감지하기</li>
</ul>
