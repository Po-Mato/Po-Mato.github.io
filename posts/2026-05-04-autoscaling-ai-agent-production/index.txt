10:T4178,<h2>들어가며</h2>
<p>A2A 프로토콜로 여러 에이전트가 협업하고, MCP로 외부 도구를 호출하는 Production AI 시스템에서 가장 큰 고통中的一个는 **"요청이 갑자기 밀려올 때 시스템이 무너지는 것"**입니다.</p>
<p>에이전트가 Concurrent하게 동작하고, 사용자가 에이전트에게 자연스럽게 여러 태스크를 동시에 밀어 넣으면서, AI inference는Compute-intensive 특성上 CPU/메모리 사용률이 예츣할 수 없이 치솟습니다.</p>
<p>이 글에서는:</p>
<ol>
<li>오토스케일링의 4가지 알고리즘 비교</li>
<li>Kubernetes HPA v2 vs KEDA 차이와 선택 기준</li>
<li>AI 에이전트 작업 부하에 최적화된 오토스케일링 아키텍처</li>
<li>실제 Production 사례에서 추출한 튜닝 패턴</li>
</ol>
<p>을 정리합니다.</p>

<h2>1. 오토스케일링의 네 가지 알고리즘</h2>
<h3>1.1 Vertical Scaling (수직 스케일링)</h3>
<p>가장 단순한 접근입니다. Pod의 resource request/limit을 높이고, 더 큰 노드로 스케줄링합니다.</p>
<pre><code class="language-yaml"># Vertical Pod Autoscaler (VPA) — Recommendation Mode
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: agent-inference-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: agent-inference
  updatePolicy:
    updateMode: "Off"  # recommendation만 보고 실제 적용은 "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: inference
      minAllowed:
        cpu: "100m"
        memory: "256Mi"
      maxAllowed:
        cpu: "8"
        memory: "16Gi"
</code></pre>
<p><strong>장점:</strong> 설정이非常简单, 노트북 한 대에서 크게 하는 것처럼 바로 적용 가능
<strong>단점:</strong> 노드 크기の上限에 도달하면 더 이상 확장 불가, 적용 순간 Pod 재시작 발생</p>
<h3>1.2 Horizontal Scaling (수평 스케일링) — HPA</h3>
<p>Pod 레플리카 수를 늘리는 방식입니다. 가장 널리 쓰이는 패턴.</p>
<pre><code class="language-yaml"># Kubernetes Horizontal Pod Autoscaler (HPA) v2
apiVersion: autoscaling.k2.x-k8s.io/v1
kind: HorizontalPodAutoscaler
metadata:
  name: agent-worker-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: agent-worker-pool
  minReplicas: 2
  maxReplicas: 50
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 5분 동안缩尺DOWN 안 함
      policies:
      - type: Pods
        value: 2
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0  # 즉시 확장 (급증 대응)
      policies:
      - type: Pods
        value: 4
        periodSeconds: 15
</code></pre>
<p><strong>핵심 포인트:</strong> <code>stabilizationWindowSeconds</code>는 <strong>스케일링 히스테리시스</strong>를 구현합니다. 너무 짧으면 thrashing(확장→축소→확장 반복), 너무 길면 반응 지연.</p>
<h3>1.3 Predictive Scaling (예측 스케일링)</h3>
<p>ML 모델로 미래 트래픽을 예측하고 미리 용량을 늘리는 방식입니다.</p>
<pre><code class="language-python"># AWS Auto Scaling — Predictive Scaling (Python boto3 예시)
import boto3

autoscaling = boto3.client('autoscaling')

# Predictive Scaling Policy 설정
autoscaling.put_scaling_policy(
    AutoScalingGroupName='agent-inference-asg',
    PolicyName='predictive-scaling',
    PolicyType='PredictiveScaling',
    PredictiveScalingConfiguration={
        'MetricSpecifications': [
            {
                'TargetResourceType': 'ComputeEnvironment',
                'CustomizedCapacityMetric': {
                    'MetricDataQueries': [
                        {
                            'Id': 'agent_active_tasks',
                            'Expression': 'agent_running_tasks / agent_max_capacity',
                            'Label': 'Agent Active Task Ratio'
                        }
                    ]
                },
                'TargetValue': 70,  # 70% 이상이면 확장
                'ScaleInCooldown': 300,
                'ScaleOutCooldown': 60
            }
        ],
        'Mode': 'ForecastAndScale',  # 예측 + 실시간 하이브리드
        'SchedulingBufferTime': 300   # 예측 시간より5분先に 확장
    }
)
</code></pre>
<p><strong>AI 에이전트 활용:</strong> 사용자가 작업을 등록하면 작업 큐에积まれる데, Predictive Scaling은 <strong>작업 제출 패턴</strong>을 학습해서 에이전트 풀 크기를 미리 늘립니다.</p>
<h3>1.4 Event-Driven Scaling — KEDA</h3>
<p>AI 에이전트 Workload에 가장 적합한 패턴입니다. 메트릭 기반이 아니라 <strong>이벤트 소스</strong> 기반으로 스케일링합니다.</p>
<pre><code class="language-yaml"># KEDA (Kubernetes Event-Driven Autoscaling)
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: agent-task-queue-scaler
spec:
  scaleTargetRef:
    name: agent-worker-pool
  pollingInterval: 5      # 5초마다 체크
  cooldownPeriod: 30      # 확장 후 30초间 축소 안 함
  minReplicaCount: 2
  maxReplicaCount: 100

  triggers:
  # 트리거 1: RabbitMQ 큐 깊이
  - type: rabbitmq
    metadata:
      host: amqp://guest:guest@rabbitmq:5672/
      queueName: agent_tasks
      activationThreshold: "10"       # 10개 이상일 때만 스케일 아웃
      threshold: "50"                  # 50개면 max까지 확장

  # 트리거 2: Prometheus — LLM API 에러율
  - type: prometheus
    metadata:
      serverAddress: http://prometheus:9090
      metricName: llm_api_error_rate
      threshold: "0.05"               # 5% 이상 에러율이면 추가 에이전트 배치
      query: |
        sum(rate(llm_api_errors_total{agent_id=~".*"}[2m]))
        /
        sum(rate(llm_api_requests_total{agent_id=~".*"}[2m]))

  # 트리&#x3C;minimax:tool_call> 3: Cron — 근무 시간 패턴
  - type: cron
    metadata:
      timezone: Asia/Seoul
      start: "0 9 * * 1-5"   # 월~금 09:00에 최소 10개副本
      end: "0 18 * * 1-5"    # 오후 6시에 원복
      desiredReplicas: "10"
</code></pre>
<p><strong>KEDA가 AI 에이전트에最强的인 이유:</strong></p>

<h2>2. HPA vs KEDA: 언제 무엇을 선택하는가</h2>
<h3>2.1 결정 매트릭스</h3>
<pre><code>                    HPA                              KEDA
─────────────────────────────────────────────────────────────────
기본 원리            메트릭 기반 스케일링             이벤트 소스 기반 스케일링
확장 속도            moderate (1~3분 딜레이)          빠름 (5초 폴링)
축소 속도            느림 (hist窗口 길다)             중간 (cooldownPeriod 튜닝)
커스텀 메트릭        가능하나 설정 복잡             非常简单 (메트릭 쿼리만)
외부 이벤트 소스     불가                             매우 다양함 (30+ 트리거)
구성 복잡도          낮음 ~ 중간                      중간
비용 효율성          배치 작업에 불리                배치 + 이벤트 드리븐에 최적
</code></pre>
<h3>2.2 Hybrid 아키텍처 (실무 권장)</h3>
<p>HPA와 KEDA를 동시에 쓰는 구성입니다.</p>
<pre><code class="language-yaml"># HPA — CPU/메모리 기반 안전망 (fallback)
apiVersion: autoscaling.k2s.io/v2
kind: HorizontalPodAutoscaler
metadata:
  name: agent-hpa-fallback
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: agent-worker-pool
  minReplicas: 2
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

---
# KEDA — 큐 기반 최적 확장 (Primary)
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: agent-keda-primary
spec:
  scaleTargetRef:
    name: agent-worker-pool
  minReplicaCount: 2
  maxReplicaCount: 100
  cooldownPeriod: 30
  triggers:
  - type: rabbitmq
    metadata:
      host: amqp://guest:guest@rabbitmq:5672/
      queueName: agent_tasks
      threshold: "20"
</code></pre>
<p><strong>이 구성의 효과:</strong></p>
<ul>
<li>KEDA가 큐 메시지 수로 빠르게 확장</li>
<li>CPU/메모리가 비정상적으로 치솟으면 HPA가 백업으로 동작
-平常時は KEDA만 동작, 이상 발생 시 HPA가 안전망 역할</li>
</ul>

<h2>4. 실전 튜닝: Production에서 얻은教训</h2>
<h3>4.1 스케일링 Thrashing 방지</h3>
<p>가장 흔한 실수는 <strong>cooldownPeriod를 너무 짧게 설정</strong>하는 것입니다.</p>
<pre><code class="language-yaml"># ❌ 잘못된 설정 — Thrashing 발생
cooldownPeriod: 5   # 5초면 확장과 축소가 반복됨

# ✅ 수정된 설정
cooldownPeriod: 300  # 5분간 축소 안 함
</code></pre>
<p><strong>경험적 규칙:</strong> 에이전트 태스크 하나의 평균 실행 시간이 30초라면, cooldownPeriod는 최소 **10배 이상(300초)**으로 설정.</p>
<h3>4.2 Panic Mode (급격 확장 방지)</h3>
<pre><code class="language-yaml">behavior:
  scaleUp:
    stabilizationWindowSeconds: 0  # 즉각 확장
  scaleDown:
    stabilizationWindowSeconds: 600 # 10분간 축소 안 함 (Panic 방지)
    policies:
    - type: Percent
      value: 10                       # 한 번에 10% 이상 축소 불가
      periodSeconds: 600
</code></pre>
<p>급격한 축소로 인한 <strong>OOM restart cascading failure</strong>를 방지합니다.</p>
<h3>4.3 Cost Cap으로 예산 관리</h3>
<pre><code class="language-yaml"># Kubernetes Resource Claim으로 비용 상한 제어
apiVersion: v1
kind: LimitRange
metadata:
  name: agent-cost-cap
spec:
  limits:
  - max:
      cpu: "64"
      memory: "128Gi"
    min:
      cpu: "100m"
      memory: "128Mi"
    type: Container
  - max:
      pods: "200"    # 전체 Pod 수로 비용 상한 관리
    type: Pod
</code></pre>

<h2>마무리</h2>
<p>AI 에이전트 시스템에서 오토스케일링은 단순한 infrastructure 설정이 아니라, <strong>서비스 신뢰성(SRE)의 핵심 구성 요소</strong>입니다.</p>
<p>핵심 정리:</p>
<ol>
<li><strong>HPA + KEDA hybrid 구성</strong>이 AI 에이전트 Workload에 가장 효과적</li>
<li><strong>KEDA의 이벤트 드리븐 확장</strong>은 큐 기반 에이전트 태스크에 최적</li>
<li><strong>Panic mode 설정</strong>으로 스케일링 Thrashing과 cascading failure 방지</li>
<li><strong>Cost cap</strong>으로 예산 초과 리스크 관리</li>
<li><strong>실시간 모니터링</strong>으로 SLO 유지 여부 확인</li>
</ol>
<p>오토스케일링을 제대로 구성하면, 에이전트 시스템은 트래픽 급증에도 안정적으로 동작하면서 불필요한 비용은最小화할 수 있습니다.</p>
