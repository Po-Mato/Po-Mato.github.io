<li><strong>자가 검토 결론</strong>: App Router의威力를 fully 활용하려면"서버와 클라이언트의 경계선"을 물리적으로 분리하는 습관이 가장 먼저 필요하다</li>
</ul>

<h2>1. 서론: 왜 Production에서만 드러나는 문제가 있는가</h2>
<p>Next.js App Router는2023년부터 본격적으로Production 환경에 들어왔다.笔者는2024~2025년 사이 50개 이상의 프로젝트를 점검하면서 하나의 공통된 패턴을 발견했다.</p>
<blockquote>
<p><strong>"데모에서는 완벽해 보이지만,Production 스케일에서 문제가 드러난다."</strong></p>
</blockquote>
<p>이는 App Router의 설계 철학 자체가 때문이다. App Router는 <strong>선언적 병렬 처리</strong>와 ** Suspense 기반 스트리밍**을 통해 TTFB와 LCP를剧的に 개선하지만, 그 mechanisms이 복잡하기 때문에 잘못된 사용이 성능을逆转시킬 수 있다.</p>
<p>이 글에서는 7가지 위험 패턴을 <strong>아키텍처적 원인 → 실제 증상 → 검증된 해법</strong> 구조로 분석한다.</p>
<hr>
<h2>2. 함정 1: "서버 컴포넌트에서 데이터를 fetching하면 무조건 빠르다"</h2>
<h3>2-1 문제의 원인</h3>
<p>가장 흔한 착각이다. 다음 코드를 보자.</p>
<pre><code class="language-tsx">// ❌ 잘못된 패턴: Server Component에서 waterfall fetching
async function Page() {
  const user = await getUser();              // 첫 번째 대기
  const orders = await getOrders(user.id);   // 두 번째 대기 (순차적!)
  
  return &#x3C;OrderList orders={orders} user={user} />;
}
</code></pre>
<p>이 패턴의 문제점은明确하다. <code>getUser()</code>가 완료될 때까지 <code>getOrders()</code>는 시작조차 하지 않는다. 네트워크 지연이 100ms + 100ms = <strong>200ms의 순차 대기</strong>가 발생한다.</p>
<h3>2-2 해결책: 병렬 데이터 fetching + Promise.all</h3>
<pre><code class="language-tsx">// ✅ 올바른 패턴: 병렬 fetching
async function Page() {
  // 동시에 시작 → 총 대기 시간 = max(100ms, 100ms) = 100ms
  const [user, orders] = await Promise.all([
    getUser(),
    getOrders()
  ]);
  
  return &#x3C;OrderList orders={orders} user={user} />;
}
</code></pre>
<h3>2-3进阶: React Server Component의cache 명시적 활용</h3>
<p>Next.js 15에서는fetch의cache 옵션이 기본으로变了:</p>
<pre><code class="language-tsx">// next: { cache: 'force-cache' }가 기본 (명시적 권장)
const user = await fetch('/api/user', {
  next: { revalidate: 3600, tags: ['user'] }
});
</code></pre>
<p><strong>핵심 규칙</strong>: 데이터의 性격에 따라 cache 전략을 분리하라.</p>
<ul>
<li><strong>정적 데이터</strong>(설정, 카탈로그): <code>revalidate: false</code> + CDN缓存</li>
<li><strong>반정적 데이터</strong>(사용자 프로필): <code>revalidate: 60</code> + ISR</li>
<li><strong>동적 데이터</strong>(주문 목록): <code>cache: 'no-store'</code></li>
</ul>

<h2>3. 함정 2: Server Actions + Optimistic Updates 없이는 UX가 뒤떨어진다</h2>
<h3>3-1 문제의 원인</h3>
<p>Server Actions는 아름다운 추상화이지만, 기본적으로 <strong>비동기 Requet-Response 모델</strong>이다. 사용자가 폼을 제출하면:</p>
<ol>
<li>서버 처리 대기 (平均 300-800ms)</li>
<li>서버 응답 수신</li>
<li>React가 상태를 갱신 → UI 업데이트</li>
</ol>
<p>이 사이 사용자에게는 <strong>무반응</strong>처럼 보인다.</p>
<h3>3-2 해결책: <code>useOptimistic</code> Hook 활용</h3>
<pre><code class="language-tsx">'use client';

import { useOptimistic } from 'react';
import { updateCartItem } from './actions';

function Cart({ items, cartId }) {
  const [optimisticItems, addOptimisticItem] = useOptimistic(
    items,
    (state, { id, quantity }) =>
      state.map(item => item.id === id ? { ...item, quantity } : item)
  );

  async function handleUpdate(id: string, quantity: number) {
    addOptimisticItem({ id, quantity }); // 即时反映
    await updateCartItem({ cartId, id, quantity });
  }

  return (
    &#x3C;ul>
      {optimisticItems.map(item => (
        &#x3C;li key={item.id}>
          {item.name} - {item.quantity}
          &#x3C;button onClick={() => handleUpdate(item.id, item.quantity + 1)}>
            +
          &#x3C;/button>
        &#x3C;/li>
      ))}
    &#x3C;/ul>
  );
}
</code></pre>
<p><strong>결과</strong>: 서버 응답을 기다리지 않고 即时 UI 반영. 실패 시 자동 롤백.</p>
<h3>3-3进阶: <code>useFormStatus</code>로 제출 상태 표현</h3>
<pre><code class="language-tsx">'use client';

import { useFormStatus } from 'react-dom';

function SubmitButton() {
  const { pending } = useFormStatus();
  
  return (
    &#x3C;button type="submit" disabled={pending}>
      {pending ? '저장 중...' : '저장'}
    &#x3C;/button>
  );
}
</code></pre>

<h2>4. 함정 3: Route Groups 잘못 사용으로 인한 Bundle 폭발</h2>
<h3>4-1 문제의 원인</h3>
<p>Next.js 15의 Route Groups <code>(folder)</code>는 URL 경로에 영향을 주지 않으면서 레이아웃을 분리하는 메커니즘이다. 하지만 이를 잘못 쓰면 문제가 생긴다.</p>
<pre><code>app/
├── (marketing)/          ← Route Group A
│   ├── layout.tsx        ← marketing 전용 layout
│   ├── page.tsx          ← /
│   └── blog/page.tsx     ← /blog
│
├── (app)/                ← Route Group B  
│   ├── layout.tsx        ← app 전용 layout
│   └── dashboard/page.tsx ← /dashboard
</code></pre>
<p>문제가 생기는 케이스:</p>
<pre><code class="language-tsx">// (marketing)/blog/page.tsx
// 이 파일은 marketing layout만 로드할 것 같지만,
// Next.js는 두 layout을 모두 병합하려고 시도한다
</code></pre>
<h3>4-2 해결책: Layout boundary를 명확하게 분리</h3>
<pre><code class="language-tsx">// ✅ 권장 구조: 명확한 관심사 분리
app/
├── (marketing)/
│   ├── layout.tsx         // 마케팅 관련 global layout
│   ├── page.tsx           // /
│   └── blog/
│       └── page.tsx       // /blog
│
├── (dashboard)/
│   ├── layout.tsx         // 대시보드 전용 layout
│   └── dashboard/
│       └── page.tsx       // /dashboard
│
└── layout.tsx             // 공통 root layout (meta, fonts만)
</code></pre>
<p><strong>중요</strong>: Root layout에는 최소한의 공통 요소만 두어라. 두 Route Group이 공유해야 할 것이 있다면, 그것은 <strong>shared component</strong>로 분리하라.</p>
<h3>4-3 Bundle 분리의 실전 검증</h3>
<pre><code class="language-bash"># 빌드 분석 명령어
npx @next/bundle-analyzer .next/server/
</code></pre>
<p>Bundle이 500KB 이상이라면, Route Group 분리가 제대로 안 된 것이다.</p>

<h2>5. 함정 4: Parallel Routes + Intercepting Routes의 디버깅 악몽</h2>
<h3>4-1 문제의 원인</h3>
<p>Instagram 스타일의 photo modal 구현을 생각해보자.</p>
<pre><code>app/
├── photo/[id]/
│   └── page.tsx           // Full page: /photo/123
│
├── feed/
│   ├── @modal/(..)photo/[id]/
│   │   └── page.tsx       // Intercepted: /feed (modal만)
│   └── page.tsx            // Feed page with slot
</code></pre>
<p>이 구조는 <strong>강력하지만</strong>, 문제가 생겼을 때 추적이 어렵다:</p>
<ul>
<li><strong>URL은 <code>/feed</code>인데 photo modal이 떠 있을 때</strong>: Router state가 두 개 존재</li>
<li><strong>브라우저 뒤로 가기</strong>: Intercepted modal → Full page 순서로 동작해야 하는데, 순서가 반대인 경우가 있음</li>
<li><strong>SEO</strong>: Modal 내용은SEO에爬虫되지 않는데,Canonical URL 설정이 까다로움</li>
</ul>
<h3>4-2 해결책: Intercepted modal의 Canonical URL 명시적 처리</h3>
<pre><code class="language-tsx">// app/feed/@modal/(..)photo/[id]/page.tsx
export default function InterceptedPhotoModal({ params }) {
  return (
    &#x3C;>
      {/* SEO를 위한 클라이언트 사이드 렌더링 */}
      &#x3C;SeoMetadata photoId={params.id} />
      
      {/* Modal UI */}
      &#x3C;dialog open className="photo-modal">
        &#x3C;PhotoContent id={params.id} />
      &#x3C;/dialog>
    &#x3C;/>
  );
}
</code></pre>
<p><strong>핵심</strong>: Intercepting Routes는 <strong>모바일 앱 경험</strong>을 위해 설계된 것이므로, 웹 SEO가 중요한 페이지에서는 사용하지 마라. 대신 <strong>Client-side navigation with portal</strong>을 고려하라.</p>

<h2>6. 함정 5: Cache Tagging + Revalidation 폭발</h2>
<h3>6-1 문제의 원인</h3>
<p>Next.js의on-demand revalidation은强大하지만, 잘못된 태그 설계는cache invalidation storm을 만든다.</p>
<pre><code class="language-tsx">// ❌ 위험한 패턴: 너무 세밀한 태그
await fetch('/api/products', {
  next: { tags: ['product', 'product-123', 'product-123-reviews', 'product-123-images'] }
});
</code></pre>
<p>제품 한 개가 갱신될 때마다 4개의cache가 invalidation 된다. 100개 제품이라면400번.</p>
<h3>6-2 해결책: 계층적 태그 설계</h3>
<pre><code class="language-tsx">// ✅ 올바른 패턴: 계층적 태그
// 제품 상세 페이지
await fetch(`/api/products/${id}`, {
  next: { tags: [`product:${id}`, 'products'] }
});

// 제품 목록 페이지
await fetch('/api/products', {
  next: { tags: ['products'] }
});

// products 태그로revlidate하면 제품 목록만 갱신
// product:123 태그로revlidate하면 해당 제품 상세만 갱신
</code></pre>
<p><strong>revlidation 함수</strong>:</p>
<pre><code class="language-ts">// app/api/revalidate/route.ts
import { revalidateTag } from 'next/cache';

export async function POST(req: Request) {
  const { tag, secret } = await req.json();
  
  if (secret !== process.env.REVALIDATE_SECRET) {
    return Response.json({ error: 'Invalid' }, { status: 401 });
  }
  
  revalidateTag(tag);
  return Response.json({ revalidated: true, tag });
}
</code></pre>

<h2>7. 함정 6: Streaming SSR + Suspense의 잘못된 사용으로 인한 CLS 악화</h2>
<h3>7-1 문제의 원인</h3>
<p>Streaming SSR은 TTFB를剧적으로 줄이지만, 잘못된 구현은 Cumulative Layout Shift(CLS)를 극적으로 악화시킨다.</p>
<pre><code class="language-tsx">// ❌ 문제: 로딩 상태가 placeholder를 제공하지 않음
async function ProductPage({ id }) {
  const product = await getProduct(id); // 500ms 대기
  
  return (
    &#x3C;div>
      &#x3C;h1>{product.name}&#x3C;/h1> {/* 이 요소가 늦게出现 → 레이아웃 시프트 */}
      &#x3C;ProductGallery images={product.images} />
    &#x3C;/div>
  );
}
</code></pre>
<h3>7-2 해결책: Skeleton + streaming으로 CLS 관리</h3>
<pre><code class="language-tsx">// app/products/[id]/page.tsx
import { Suspense } from 'react';

export default function ProductPage({ params }) {
  return (
    &#x3C;div>
      {/* 즉시 렌더링 → 로딩 상태 placeholder 제공 */}
      &#x3C;h1>&#x3C;Skeleton width="300px" height="32px" />&#x3C;/h1>
      
      &#x3C;Suspense fallback={&#x3C;GallerySkeleton />}>
        &#x3C;ProductGallery id={params.id} />
      &#x3C;/Suspense>
    &#x3C;/div>
  );
}

// 서버 컴포넌트: gallery만 지연 로딩
async function ProductGallery({ id }) {
  const images = await getProductImages(id); // 느린 데이터
  return &#x3C;ImageGrid images={images} />;
}
</code></pre>
<p><strong>원리</strong>: 클라이언트가 먼저 보는 영역은 skeleton으로 즉시 제공하고, 느린 데이터 영역만 Suspense로 감싸면 레이아웃 시프트가 발생하지 않는다.</p>

<h2>8. 함정 7: Edge Runtime 선택의 함정</h2>
<h3>8-1 문제의 원인</h3>
<p>Next.js 15에서 Edge Runtime은매우 매력적이다. 全球分布으로 50ms 이내 TTFB 가능.</p>
<pre><code class="language-tsx">// app/api/edge-route/route.ts
export const runtime = 'edge';

export async function GET(req: Request) {
  // 이 코드는 V8 isolates에서 실행됨
  const token = req.headers.get('authorization');
  // Node.js API인 crypto.subtle는 사용 불가!
  const hash = await crypto.subtle.digest(/* ... */); // 일부만 가능
}
</code></pre>
<p><strong>Edge Runtime의 제한 사항</strong>:</p>
<ul>
<li><code>fs</code>, <code>path</code> 모듈 사용 불가</li>
<li><code>Buffer</code> 생성 불가 (Web APIs만 사용 가능)</li>
<li><code>setTimeout</code>/<code>setInterval</code>이 없음 (Cold Start 최적화를 위해 제거)</li>
<li>Node.js Built-in module 중 일부는 불가</li>
</ul>
<h3>8-2 해결책: Runtime 선택 기준</h3>
<pre><code class="language-tsx">// ✅ Edge-compatible한 코드
export const runtime = 'edge';

export async function GET(req: Request) {
  const url = new URL(req.url);
  
  // Web API만 사용
  // Headers, Request, URL은 모두 Web Standard
  // crypto.subtle는 사용 가능
  
  return Response.json({ ok: true });
}

// ❌ Node.js가 필요한 경우 → Node.js runtime 명시
export const runtime = 'nodejs';

export async function GET(req: Request) {
  const fs = await import('fs'); // Node.js 모듈
  const data = fs.readFileSync('./data.json', 'utf-8');
  return Response.json(JSON.parse(data));
}
</code></pre>
<p><strong>결론</strong>: Edge는 데이터 변환, 인증, A/B 테스트 분기에만 사용하라. 데이터베이스 접근이 필요한 API Routes는Standard(Node.js) 또는 Serverless Runtime을 사용하라.</p>

<h2>10. 결론</h2>
<p>Next.js 15 App Router는 올바르게 사용하면 매우强大的 도구다. 그러나 그 мощ함은 complexity를 수반한다. **"서버와 클라이언트의 경계선을 물리적으로 분리하는 습관"**이 가장 중요하다.</p>
